## Context

v0.17.0 introduced hash-based smart-diff via `.sdd-meta.json` but scoped provenance tracking to **template agents + `AGENTS.md`** only. Other stack-adapted or customization-prone files remain on v0.16.10 behavior: wholesale-recopy or `isStandardModified` compare. This plan closes that gap *partially* for v0.17.1 (see "Scope" below for the narrow selection and "Out of scope" for the deferred remainder).

**Motivating incident**: foodXPlorer upgrade from v0.16.9 to v0.17.0 on 2026-04-14. Two real defects surfaced:

1. The scanner's `detectBackend` / `detectFrontend` only read the top-level `package.json`. Monorepos (npm/pnpm/yarn workspaces) store actual deps in workspace `package.json` files, so the scanner returned `{ detected: false }` for foodXPlorer and `adaptAgentsMd` fell back to the template literal `Backend patterns (DDD, Express, Prisma)` instead of the real `(layered, Fastify, Prisma, Kysely)`.
2. The post-upgrade CLI message advertised provenance tracking as "v0.17.0 will fix" even when the user was already running v0.17.0. Misleading copy.

A third issue (Gemini Q10 from v1.0 review): when `scan.backend.detected === true` but `scan.backend.framework === null` (e.g., scanner detects an ORM but no framework), `adaptAgentsMd`'s `parts.filter(Boolean).join(', ')` silently drops the null framework, producing e.g. `Backend patterns (layered, , Prisma)` with a phantom comma or malformed output. Worth fixing in the same scanner pass.

## Scope (three features in one patch release)

### Feature 1 — Provenance for standards + workflow core (10 files, tight scope)

**Goal**: eliminate cross-version false-positive preserve warnings on the 10 files most critical to SDD DevFlow correctness: the 4 standards `.mdc` files, the `development-workflow` skill entry point (`SKILL.md` × 2 tools), the per-feature artifact template (`ticket-template.md` × 2), and the checkpoint checklist (`merge-checklist.md` × 2).

**Files added to provenance coverage**:

| POSIX path | Currently handled by | Why include in v0.17.1 |
|---|---|---|
| `ai-specs/specs/base-standards.mdc` | `isStandardModified` compare | Stack-independent but frequently customized by users |
| `ai-specs/specs/backend-standards.mdc` | `isStandardModified` compare + `adaptBackendStandards()` (Mongoose pre-gen) | Stack-adapted + frequently customized |
| `ai-specs/specs/frontend-standards.mdc` | `isStandardModified` + project-type pruning | Frequently customized |
| `ai-specs/specs/documentation-standards.mdc` | imperative project-type branch in `stack-adaptations.js` | Already adapted; benefits from provenance |
| `.claude/skills/development-workflow/SKILL.md` | wholesale-recopy + stack-adaptations rule table (Zod) | Entry point for the 6-step workflow, B+D checkpoint mechanism |
| `.claude/skills/development-workflow/references/ticket-template.md` | wholesale-recopy + stack-adaptations rule table (Zod) | Per-feature artifact template, teams customize for their PR/tracking format |
| `.claude/skills/development-workflow/references/merge-checklist.md` | wholesale-recopy (no adaptation) | **Core of the B+D checkpoint pattern** — this is the file that survives context compaction per user-documented invariant |
| `.gemini/skills/development-workflow/SKILL.md` | same as Claude equivalent | Mirror |
| `.gemini/skills/development-workflow/references/ticket-template.md` | same | Mirror |
| `.gemini/skills/development-workflow/references/merge-checklist.md` | same | Mirror |

Total: **10 new paths** under provenance. **Inclusion rule** (revised v1.2 per Codex Q1+Q2): *all 4 standards files in `ai-specs/specs/` (regardless of whether they currently use `isStandardModified` compare, imperative stack-adaptation, or identity — having all standards under provenance is the principled choice; omitting one because its adapter is different is arbitrary)* PLUS *the 3 workflow-core files that implement the B+D checkpoint pattern identified in the user's project memory as "el problema central de la librería" (SKILL.md + ticket-template.md + merge-checklist.md, × 2 tools)*. Together: 4 + (3 × 2) = 10 files. This rule closes the Codex M1 option 2 deferred item for all standards files in one go, and protects the three files that make context-compaction-survivable checkpoints work.

**Decision tree** (identical to v0.17.0 for agents):

1. File missing / `--force-template` → unconditional write + hash update.
2. Stored hash exists → compare current file hash.
   - **Match** → pristine, replace with newly adapted content, update hash.
   - **Mismatch** → customized, preserve, `.new` backup, DO NOT update hash (Codex M1 invariant carried forward).
3. No stored hash → fallback content-compare against the FULL adapted target.
   - For `backend-standards.mdc`: adapted target = `adaptBackendStandards(rawTemplate, scan, config)` output (keeps the existing dedicated function — see D1).
   - For the other 9 files: adapted target = `applyStackAdaptationsToContent(raw, posixPath, scan, config)`.
   - Match → replace + hash. Mismatch → preserve + `.new` backup.

**Removed/deprecated**:
- `isStandardModified` is removed entirely. Its main-path compare is replaced by the hash decision tree above; its fallback-path wrapper is renamed (see next bullet).
- New helper `normalizedContentEquals(a, b)` in `lib/meta.js` (co-located with `normalizeForCompare`) wraps `normalizeForCompare(a) === normalizeForCompare(b)`. File-agnostic name reflects actual semantics. Codex round-1 Q9 wanted `isStandardModified` deleted (it was specific to standards); Gemini round-1 Q9 wanted a named helper for fallback-path readability; Codex round-2 Q11 pointed out the kept-name-as-wrapper compromise was vestigial (the name still implied standards-specific policy). Final resolution: keep the abstraction, rename it to something file-agnostic, move to `lib/meta.js` for colocation with its only dependency.

### Feature 2 — Monorepo-aware scanner + `framework=null` formatting fix

**Goal**: make `detectBackend` / `detectFrontend` work correctly on npm/yarn workspaces monorepos, and fix the `adaptAgentsMd` formatting edge case when `scan.backend.framework === null`.

**Current behavior** (bug):

```javascript
// lib/scanner.js:64
function detectBackend(dir, pkg) {
  const deps = getAllDeps(pkg);  // ← only reads root package.json
  // ...
}
```

For foodXPlorer: root `package.json` has `"workspaces": ["packages/*"]` and no direct backend deps. Result: `detected: false`, `adaptAgentsMd` uses the template literal, `(DDD, Express, Prisma)` appears as unsubstituted placeholder.

**Proposed behavior**:

1. `scan()` runs root-level `detectBackend`/`detectFrontend` as today (top-level deps may be hoisted in some monorepos).
2. If `scan.isMonorepo && !scan.backend.detected`, enumerate workspace `package.json` files:
   - Read `pkg.workspaces` (array form or `{ packages: [...] }` object form — npm + yarn).
   - **Glob expansion with deterministic ordering** (D2): preserve declared pattern order as the outer key; within each pattern, sort matched paths lexically (case-sensitive, Unicode codepoint order). This is the invariant Codex + Gemini both flagged in v1.0 — glob expansion order must not depend on filesystem iteration order.
   - For each resolved workspace dir, read its `package.json` and call `detectBackend(wsDir, wsPkg)`.
3. **Aggregation**: first workspace whose detection returns `detected: true` wins. Store its result in `scan.backend` and record `scan.backend.workspaceSource = "<workspace relative path>"` for debuggability (so users running `--doctor` or `--diff` can see *which* workspace contributed the detection).
4. Frontend aggregation mirrors backend.
5. **Explicit non-goal**: do NOT try to pick "the best" workspace when multiple have backend stacks. Use first match. Users with exotic layouts can customize `AGENTS.md` directly (which is preserved by smart-diff anyway). Codex Q11 flagged this; accepted as explicit non-goal with the `workspaceSource` breadcrumb for diagnostics.
6. **pnpm workspaces** (`pnpm-workspace.yaml`): DEFERRED TO v0.17.2. See "Out of scope" below. Codex Q10 was correct — promising "pnpm + yarn + npm" in v1.0 was over-scope. v0.17.1 narrows to npm/yarn `package.json#workspaces`.

**Framework=null formatting fix** (new in v1.1, from Gemini Q10):

Current `adaptAgentsMd` code path when `scan.backend.detected === true` but `scan.backend.framework === null` (e.g., scanner detected an ORM but no framework binding):

```javascript
// lib/init-generator.js (approx)
const parts = [
  patternLabelFor(scan.srcStructure.pattern),
  scan.backend.framework,  // ← null
  scan.backend.orm,
].filter(Boolean);
// parts = ["layered", "Prisma"] — framework dropped silently
```

After Feature 2's scanner fix, monorepo projects will more often have `framework !== null`, BUT the edge case still exists for non-monorepo projects with ORM-only detection. Fix: instead of relying on `.filter(Boolean)` which swallows `null`, emit a cleaner message. Two options:

- **Option F2a**: fall through to `null` check → emit `parts = ["layered", "Prisma"]` → render as `(layered, Prisma)` (current `.filter(Boolean)` behavior). This is actually fine — the output is valid markdown, just missing one element. **Not a bug, a degraded but correct fallback.**
- **Option F2b**: require framework to be non-null OR skip the whole adaptation (template literal stays).

**Recommendation**: keep current `.filter(Boolean)` behavior (Option F2a) but add a doctor check #15 extension: warn when `AGENTS.md` contains `Backend patterns (...)` where `...` has fewer than 2 entries, prompting user to verify scanner detection. This is observability, not a behavior change.

## Design Decisions (revised per review)

### D1 — Dedicated adapters for standards (REVISED per Codex Q3 + Gemini Q3)

**v1.0 recommendation**: extend `stack-adaptations.js` rule table to cover standards (Option A).

**v1.1 recommendation (reversed)**: keep `adaptBackendStandards()` and any other existing dedicated adapters AS-IS. Hash their OUTPUT, not the rule-table output.

Rationale: `adaptBackendStandards` currently pre-generates Mongoose best-practice paragraphs based on the detected ORM. This is richer than find/replace and does not fit the flat rule-table model without introducing a new rule type (`insertBefore`/`insertAfter`). Both reviewers converged on "refactoring the adapter architecture is out of scope for a patch release — just wire provenance around the existing function". Accepted.

The decision tree's step 3 ("fallback compare against adapted target") uses the appropriate adapter per file:
- `backend-standards.mdc` → `adaptBackendStandards(raw, scan, config)`
- `frontend-standards.mdc` → `adaptFrontendStandards(raw, scan, config)` if it exists, else `raw`
- `base-standards.mdc` → `raw` (no adaptation)
- `documentation-standards.mdc` → `applyStackAdaptationsToContent(raw, ..., scan, config)` (existing imperative branch)
- `SKILL.md`, `ticket-template.md`, `merge-checklist.md` → `applyStackAdaptationsToContent(raw, ..., scan, config)` (existing rule-table path)

This is a small refactor: `upgrade-generator.js` needs a dispatch table mapping POSIX path → adapter function. **v1.2 clarification (Codex round-2 Q5)**: the existing v0.17.0 smart-diff code paths are NOT a general dispatch table — the agents path is a `for-of` loop over `TEMPLATE_AGENTS` calling `adaptAgentContentString`, and the `AGENTS.md` path is a hardcoded block calling `adaptAgentsMd`. The v0.17.1 dispatch table is NEW structure, not a replication of existing code. **Acceptance criterion**: introduce the dispatch table only for the 10 new files. Agents + `AGENTS.md` handling remains byte-identical to v0.17.0 (no regression in existing paths — verified by regression scenarios 53-58b plus the new full-upgrade-idempotency scenario 67).

### D2 — Deterministic monorepo glob enumeration (REVISED per Codex Q4 + Gemini Q4)

**v1.0 recommendation**: "first workspace match wins, declaration order".

**v1.2 recommendation** (revised): explicit three-level ordering — (outer) preserve the declaration order of patterns in `pkg.workspaces`; (inner) for each pattern, sort matched paths lexically by Unicode codepoint; (final) **deduplicate the flat list by normalized relative workspace path, retaining the first occurrence** (Codex round-2 Q7 + Gemini round-2 Q7).

Example 1 — typical layout: given `"workspaces": ["packages/*", "apps/*"]`:
1. Expand `packages/*` → `[packages/api, packages/bot, packages/shared, packages/web]` (sort lexically).
2. Expand `apps/*` → `[apps/admin, apps/landing]` (sort lexically).
3. Flatten + dedupe → `[packages/api, packages/bot, packages/shared, packages/web, apps/admin, apps/landing]` (no duplicates).
4. Iterate in that order, call `detectBackend`, stop at first `detected: true`.

Example 2 — overlapping patterns: given `"workspaces": ["packages/*", "packages/api"]`:
1. Expand `packages/*` → `[packages/api, packages/bot, packages/shared]`.
2. Expand `packages/api` → `[packages/api]`.
3. Flatten → `[packages/api, packages/bot, packages/shared, packages/api]`.
4. Dedupe by normalized path, first occurrence wins → `[packages/api, packages/bot, packages/shared]`. The second `packages/api` is dropped.
5. Iterate, stop at first `detected: true`. No wasted `detectBackend` call.

Invariant: identical `package.json` layout on two different filesystems (macOS HFS+ vs Linux ext4 vs Windows NTFS) produces the SAME detection result. Smoke scenarios 62 (basic detection) and 63b (deterministic ordering across fake fs orderings) enforce this; an additional sub-case in 62 exercises overlapping patterns for dedup.

### D3 — Meta read compatibility (CORRECTED per Codex Q5 + Gemini Q5 + empirical verification)

**v1.0 claim** (INCORRECT): "v0.17.0's `readMeta` will ignore extra keys gracefully."

**v1.1 corrected claim**: empirical verification against `lib/meta.js:130-167` + `pruneExpectedAbsent` shows:

- `readMeta` does NOT strictly validate key membership — it reads all entries, drops malformed ones (non-string values, wrong hash shape), and returns the cleaned object.
- `pruneExpectedAbsent` (called by install + upgrade paths) iterates the hashes map and DROPS any key not in `expectedSmartDiffTrackedPaths(aiTools, projectType)`.
- In v0.17.0's `expectedSmartDiffTrackedPaths`, only template agents + `AGENTS.md` are expected. The 10 new v0.17.1 paths are NOT.

**Consequence**: if a user runs v0.17.0 against a project that was previously upgraded with v0.17.1, the v0.17.1 extra hashes are **silently pruned** from the written-back meta. The meta file loses them. A subsequent re-upgrade to v0.17.1 then hits the fallback path for the 10 files (no stored hash), regenerating them if pristine or preserving them if customized.

**Rollout characterization** (renamed from "Downgrade safety" per Codex Q7):

- **Meta read compatibility (limited)**: v0.17.0 does not crash, corrupt, or error on a v0.17.1-written `.sdd-meta.json`. It reads the file successfully and selectively uses only the subset of keys it recognizes. Subsequent v0.17.0 runs actively PRUNE the v0.17.1-specific keys.
- **Downgrade path is lossy** (not corruption). Re-upgrading to v0.17.1 triggers the fallback compare path for all 10 newly-covered files — same as a fresh v0.17.1 upgrade from v0.17.0. **No data loss**, just one round of re-verification.
- **Acceptable trade-off** for a library that is append-only in versioning semantics (downgrade is not a primary use case).
- **Alternative considered and rejected**: bumping `CURRENT_SCHEMA_VERSION` from 1 to 2. Rejected because v0.17.0's `readMeta` falls back to null (full fallback path for ALL files) when `schemaVersion > CURRENT`, which is strictly worse than selective pruning.

**Invariant**: v0.17.1's `expectedSmartDiffTrackedPaths` is a strict superset of v0.17.0's. The 10 new paths are additive; no existing paths change. Verified by scenario 65.

### D4 — Doctor check #15 extension (unchanged from v1.0)

## Invariants

**Carried over from v0.17.0** (must not regress):

- **Codex M1**: preserved files keep their old hash (never updated on preserve).
- **Codex round 2 P1**: `--init` excludes pre-existing user files from the hashed set.
- **Gemini M1**: fallback path applies stack adaptations (and other relevant adapters) in memory to the comparison target.
- **Gemini M3**: pruning uses expected-presence, not on-disk-presence.
- **Normalization-safe**: `normalizeForCompare` strips CR, trailing whitespace, leading/trailing blank lines. **v1.2 enforcement scope (Codex + Gemini round-2 Q12)**: scenario 68 parameterizes over ALL smart-diff-tracked files (v0.17.0 agents + `AGENTS.md` + v0.17.1's 10 new files), not just the v0.17.1-new set. Prior releases never exercised CRLF explicitly in the smoke suite for v0.17.0 files; v0.17.1 closes that latent gap.

**New for v0.17.1**:

- **Scoped hash coverage**: `expectedSmartDiffTrackedPaths` is a strict superset of v0.17.0 (10 new paths added, no existing paths changed).
- **Deterministic monorepo enumeration**: workspace expansion preserves declared-pattern order outer, lexical Unicode-codepoint order inner, then **deduplicates by normalized relative workspace path (first occurrence wins)**. Filesystem iteration order does not affect detection.
- **Scanner additive** (v1.2 new — Codex + Gemini round-2 Q9): v0.17.1's `detectBackend` / `detectFrontend` are strict supersets of v0.17.0's. For single-package projects (no workspaces), v0.17.1 produces the EXACT same `scan.backend.*` / `scan.frontend.*` as v0.17.0 (byte-identical). For monorepos where v0.17.0 returned `{ detected: false }`, v0.17.1 may return `{ detected: true, ...workspaceInfo }`. v0.17.1 **never produces LESS information** than v0.17.0 for the same input. Consequence: `adaptAgentsMd` output with v0.17.1 scan is equal-or-richer than v0.17.0 for the same fixture → pristine `AGENTS.md` files on upgrade get equal-or-better content → **no regression risk on pristine upgrade path**. Locked in by scenario 63c.
- **Adapter dispatch table**: `upgrade-generator.js` dispatches each tracked path to its appropriate adapter function. `backend-standards.mdc` → `adaptBackendStandards`; workflow-core files → `applyStackAdaptationsToContent`; others → identity. Agents + `AGENTS.md` handling remains byte-identical to v0.17.0.
- **Meta read compatibility (limited)**: v0.17.0 reading a v0.17.1-written meta does not crash, does not corrupt, does not error. It actively prunes v0.17.1-specific keys on write-back (by design). Downgrade path is lossy but recoverable via re-upgrade.
- **Normalization resilience across all tracked files**: scenario 68 parameterized over ALL smart-diff tracked files (~21 paths: v0.17.0 agents + `AGENTS.md` + v0.17.1's 10 new files). Composite drift per path tests CRLF + trailing whitespace + leading/trailing blank lines in one shot.

## Rollout / Compatibility

- **Version bump**: `0.17.0` → `0.17.1`. Patch semver: additive provenance coverage + scanner bug fix + cosmetic copy fix. No breaking changes.
