# v0.17.3 — Workspace-aware auxiliary detection

> **Plan v1.2** | 2026-04-16 | Target: v0.17.3 patch release
>
> Extends v0.17.1/v0.17.2's workspace-awareness from `detectBackend` / `detectFrontend` core framework detection to four auxiliary detection paths that currently only run against the project root: `detectLanguage`, `detectTests`, `detectArchitecture`, and `detectFrontend`'s auxiliary fields (styling / components / state).
>
> **Round 1 cross-model review — Codex REJECT + Gemini REJECT** (both models agreed on the CRITICAL). 2 CRITICAL + 6 IMPORTANT findings across both models, 1 empirical inaccuracy. Plan v1.1 addresses all findings. Key architectural change: introduce a "primary workspace per side" concept that is always populated for monorepos, independent of v0.17.2's `workspaceSource` (which only fires when root lacks a framework). This fixes the real failure mode: fx root has `"next": "^14.2.29"` in deps, so v0.17.2 enumeration skipped frontend entirely → `frontend.workspaceSource === null` → v1.0's gate would have been a no-op for fx's frontend aux. Plan v1.1's single enumeration pass always establishes `primaryBackendWs` / `primaryFrontendWs` regardless of where framework was detected.

## Context

During the v0.17.2 empirical validation against foodXPlorer on 2026-04-15, the `--upgrade` produced a `backend-standards.mdc` with these cosmetic degradations:

- `Runtime: Node.js with JavaScript` — should be `TypeScript` (fx has TS throughout `packages/api` + `packages/web` + friends)
- `Testing: Not configured` — should be `Vitest` (fx has vitest configs in `packages/api`, `packages/bot`, `packages/scraper`)
- `Architecture: Custom` — should be `DDD layered` (fx has `packages/api/src/{domain,application,infrastructure}/` structure)

And `AGENTS.md` post-upgrade emitted a doctor check #14 WARN:

- `Frontend patterns has only 1 entry (Next.js) — scanner detection may be incomplete`

because `frontend.styling` / `frontend.components` / `frontend.state` all resolved to `null` even though `packages/web/package.json` declares `tailwindcss`.

**Not a correctness bug** — the library still ships correct code and preserves user customizations. But the output quality for monorepos is visibly lower than for single-package projects. This plan closes that gap, bringing the monorepo UX to parity with single-package.

Ticket origin: `dev/ROADMAP.md` `### Known follow-ups` item 1 (post v0.17.2 cleanup commit `bc7f22c`).

## Scope (single feature, patch release)

Four auxiliary detection functions in `lib/scanner.js`, currently root-only, become workspace-aware. **Revised per Round 1 review**: rather than gating on v0.17.2's `workspaceSource`, we always enumerate workspaces for monorepos and establish a "primary workspace per side" independently. This is the only way to fix fx-shaped monorepos where root package.json carries the framework dep (e.g., fx has `"next": "^14.2.29"` hoisted) — v0.17.2's promotion gate `!framework` never fires there, so `workspaceSource` stays null.

### In scope

1. **`detectLanguage(dir)` — scalar return** (`'typescript'` | `'javascript'`). Currently reads root `tsconfig.json` + root `src/*.ts`. Promote from **primary backend workspace** when monorepo.

2. **`detectArchitecture(dir, pkg)` — object return** (`{ pattern, dirs, hasControllers, hasRoutes, hasModels, hasServices, hasDomain, hasMiddleware, hasFeatures, hasHandlers }`). Currently reads via `findSrcRoot(dir)` (root-only). Promote from **primary backend workspace** when monorepo.

3. **`detectTests(dir, pkg)` — object return with split storage + per-field merge**. Currently a single `scan.tests` object is consumed by `adaptBackendStandards` AND `adaptFrontendStandards`, but in a monorepo the backend and frontend sides can use different frameworks (e.g., fx: `packages/api` uses vitest, `packages/web` uses jest). We introduce `scan.backendTests` + `scan.frontendTests`. **Crucial** (Gemini round-1): the workspace promotion must be **per-field merge**, not object replacement — root-level E2E config (Playwright/Cypress typically installed at root in monorepos) must be preserved when the workspace promotes unit-test framework. `scan.tests` is kept unchanged at the root-level for tracker, but `init-wizard.js` summary is migrated to fall back to backend/frontend tests (see "Implementation Plan" Phase 2 below).

4. **`detectFrontend` auxiliary fields** (`styling`, `components`, `state`). The core framework detection is workspace-aware via v0.17.2 **only when root lacked the framework**. For fx, `next` is at root → v0.17.2 never enumerated frontend workspaces → auxiliary fields (which v0.17.2's `Object.keys` loop WOULD have copied if it ran) stay at root-detection values (null). Plan v1.1 fixes this by always enumerating in monorepos and promoting aux fields from the primary frontend workspace.

### Out of scope

- **Multi-workspace aggregation of test frameworks.** If a monorepo has two backend workspaces each with different test frameworks, we take the first one detected (winner = first workspace where `detectTests` returns a non-default `framework`). Same behavior as v0.17.1/v0.17.2 for backend framework.
- **Deprecation of `scan.tests`.** Kept as-is for backward compat with tracker's F001 recommendation (`init-generator.js` lines 236-238, 852) and init-wizard's console output (`init-wizard.js` lines 44-45). Marked as "legacy root-level" in the docblock.
- **Separate `scan.backendLanguage` / `scan.frontendLanguage`.** Language is a single field. We promote from backend workspace. If a monorepo has a TS backend and JS frontend (or vice versa), scan.language reflects backend. Rationale: backend side is the primary consumer (adaptBackendStandards uses scan.language for the Runtime line; adaptFrontendStandards uses it for the Technology Stack Language line but this is a minor precision loss in mixed-language monorepos, not a correctness issue).
- **Doctor check #14 extension.** Stays as-is — after this fix, fx's doctor WARN for sparse Frontend patterns will no longer fire (because styling='Tailwind CSS' is populated). No check change needed.
- **`listRootDirs`, `detectExistingDocs`, `detectGitBranch`.** These are intentionally project-root-scoped (project tree listing, doc discovery, git branch) and have no workspace-level analog.
- **`detectMonorepo`.** Already detects monorepo shape; no behavior change.

## Design Decisions

### D1 — Scalar merge for `detectLanguage`

The v0.17.2 merge pattern uses `Object.keys(wsBackend)` iteration with a non-null guard. `detectLanguage` returns a **string**, not an object. Iterating `Object.keys('typescript')` would enumerate character indices — broken.

**Decision**: special-case language as a scalar merge:

```js
const wsLanguage = detectLanguage(wsAbs);
if (wsLanguage === 'typescript') language = 'typescript';
// Only promote TypeScript over JavaScript (the default); never demote TS → JS.
```

Rationale: `'javascript'` is the default returned when root has no `tsconfig.json` and no root `src/*.ts`. If the workspace also returns `'javascript'`, no change needed. If it returns `'typescript'`, promote. The inverse (root TS, workspace JS) can't happen in practice: if root has a tsconfig, detectLanguage already returned TS before we enter this block, and we don't revisit.

### D2 — Boolean truthiness guard for `detectArchitecture`

`detectArchitecture` returns an object with several boolean flags (`hasDomain`, `hasControllers`, `hasServices`, etc.). The v0.17.2 non-null guard (`value !== null && value !== undefined`) does NOT protect against `false` overwriting `true`:

```js
// NAIVE (broken for booleans):
for (const field of Object.keys(wsArchitecture)) {
  if (wsArchitecture[field] !== null && wsArchitecture[field] !== undefined) {
    srcStructure[field] = wsArchitecture[field];  // false overwrites true!
  }
}
```

**Decision**: field-specific merge logic per return shape:

```js
const wsArchitecture = detectArchitecture(wsAbs, wsPkg);
// Promote pattern if workspace found a known pattern (not 'unknown')
if (wsArchitecture.pattern !== 'unknown') srcStructure.pattern = wsArchitecture.pattern;
// Promote dirs array if workspace found any (non-empty)
if (wsArchitecture.dirs && wsArchitecture.dirs.length > 0) srcStructure.dirs = wsArchitecture.dirs;
// Promote boolean flags only from false → true (never demote)
for (const field of ['hasControllers','hasRoutes','hasModels','hasServices','hasDomain','hasMiddleware','hasFeatures','hasHandlers']) {
  if (wsArchitecture[field] === true) srcStructure[field] = true;
}
```

Rationale: a boolean flag means "this pattern was observed somewhere"; if the root or workspace sees it, preserve. The OR-merge semantics match what a user expects when reading `hasDomain: true` in the scan output.

### D3 — Split `scan.tests` into backendTests + frontendTests with PER-FIELD merge

**Problem**: a single `scan.tests` field is consumed by both `adaptBackendStandards` and `adaptFrontendStandards`. In mixed-framework monorepos (e.g., fx: api=vitest, web=jest), whichever workspace we promote from will be wrong for the other side.

**Decision (REVISED per Gemini round-1 CRITICAL)**: add two new fields, keep the legacy one, **and do per-field merge to preserve root-level E2E config**:

```js
// In scanner.js scan()
return {
  // ...
  tests: rootTests,              // LEGACY: root-level detection (unchanged semantics)
  backendTests: backendTests,    // NEW: per-field-merged from primary backend workspace
  frontendTests: frontendTests,  // NEW: per-field-merged from primary frontend workspace
  // ...
};
```

**Why per-field merge, not object replacement**: monorepos commonly install Playwright/Cypress at root (one E2E suite covers multiple workspaces) while unit-test frameworks (Vitest/Jest) live per-workspace. Naive `backendTests = wsTests` would erase `rootTests.e2eFramework = 'playwright'` when promoting `wsTests.framework = 'vitest'`. Merge strategy:

```js
function mergeWorkspaceTests(rootTests, wsTests) {
  const merged = { ...rootTests };
  // Workspace wins for unit framework if it detected one (framework !== 'none')
  if (wsTests.framework !== 'none') {
    merged.framework = wsTests.framework;
    merged.hasConfig = wsTests.hasConfig || rootTests.hasConfig;
  }
  // Counts: workspace authoritative when it detected anything.
  // REVISED v1.2 (Gemini round-2 CRITICAL): the naive `wsTests.testFiles > rootTests.testFiles`
  // condition is dead code because `countFilesRecursive` walks up to depth 6 from projectDir
  // and traverses into `packages/*`, so rootTests.testFiles ALREADY aggregates workspace counts
  // and is always >= wsTests.testFiles. The workspace-authoritative metric is "workspace saw
  // any test files" (> 0), not "more than root".
  if (wsTests.testFiles > 0) {
    merged.testFiles = wsTests.testFiles;
    merged.testDirs = wsTests.testDirs;
    merged.estimatedCoverage = wsTests.estimatedCoverage;
  }
  // E2E: prefer workspace if it set one; otherwise preserve root
  if (wsTests.e2eFramework) merged.e2eFramework = wsTests.e2eFramework;
  return merged;
}
```

This is safe: a monorepo with root Playwright + workspace Vitest gets `{ framework: 'vitest', e2eFramework: 'playwright', testFiles: <workspace-count>, ... }` — both signals preserved, and the count reflects the primary workspace (semantically meaningful for adapt-standards "Testing" line).

Then migrate:
- `adaptBackendStandards` (`init-generator.js:461-462`): `scan.tests.framework` → `scan.backendTests.framework`
- `adaptFrontendStandards` (`init-generator.js:575`): `scan.tests.framework` → `scan.frontendTests.framework`
- `init-wizard.js:44-48` summary: fall back to backendTests/frontendTests when rootTests.framework is 'none' but a side has a detected framework. Prevents "None detected" UX regression for fx-shaped monorepos (Gemini round-1 IMPORTANT).

Consumers that stay on `scan.tests`:
- Tracker F001 recommendation (`init-generator.js:236-238, 852`) — project-level signal, reasonable to use OR of rootTests/backendTests/frontendTests `framework === 'none'` as the "low-coverage heuristic" trigger. Addressed in Phase 2.

**Single-package invariant**: when `!isMonorepo`, no workspace enumeration runs and `scan.backendTests === scan.frontendTests === scan.tests` (reference-equal). Scenario 78 asserts this.

### D4 — Auxiliary frontend fields — **independent enumeration**, NOT `workspaceSource` gate

**REVISED per Codex + Gemini round-1 CRITICAL**. Plan v1.0 said "promote from frontend workspace when monorepo and `frontend.workspaceSource` is set". This is broken for the exact case that motivated v0.17.3 (fx root has `"next": "^14.2.29"` → v0.17.2 skipped enumeration → `frontend.workspaceSource === null` → v1.0's aux promotion would never fire).

Plan v1.1: we always enumerate monorepo workspaces for aux detection, independently of v0.17.2's framework promotion. See D5.

### D5 (NEW) — Single-pass enumeration with "primary workspace per side"

**Problem** (Codex + Gemini CRITICAL): v0.17.2's enumeration only fires when the root lacks a framework. If root has the framework hoisted, no enumeration → no `workspaceSource` → no handle on which workspace holds the "real" backend or frontend code. v0.17.3 aux detection needs a workspace handle regardless of where framework came from.

**Decision**: replace the two-phase design (v0.17.2 promotion gated on `!framework`, v0.17.3 aux gated on `workspaceSource`) with a **single enumeration pass** that always runs in monorepos and always establishes `primaryBackendWs` + `primaryFrontendWs`. Merged v0.17.2 promotion and v0.17.3 aux into the same loop.

Pseudocode:

```js
if (isMonorepo) {
  const workspaces = enumerateWorkspaces(projectDir, pkg);
  let primaryBackendWs = null;
  let primaryFrontendWs = null;

  for (const wsRel of workspaces) {
    const wsAbs = path.join(projectDir, ...wsRel.split('/'));
    const wsPkg = readPackageJson(wsAbs);

    if (!primaryBackendWs) {
      const wsBackend = detectBackend(wsAbs, wsPkg);
      if (wsBackend.framework) {
        primaryBackendWs = wsRel;
        // v0.17.2 behavior — only merge if root didn't already have framework
        if (!backend.framework) {
          for (const field of Object.keys(wsBackend)) {
            if (wsBackend[field] !== null && wsBackend[field] !== undefined) {
              backend[field] = wsBackend[field];
            }
          }
          backend.workspaceSource = wsRel;
        }
      }
    }

    if (!primaryFrontendWs) {
      const wsFrontend = detectFrontend(wsAbs, wsPkg);
      if (wsFrontend.framework) {
        primaryFrontendWs = wsRel;
        if (!frontend.framework) {
          for (const field of Object.keys(wsFrontend)) {
            if (wsFrontend[field] !== null && wsFrontend[field] !== undefined) {
              frontend[field] = wsFrontend[field];
            }
          }
          frontend.workspaceSource = wsRel;
        }
      }
    }

    if (primaryBackendWs && primaryFrontendWs) break;
  }

  // v0.17.3 aux detection — runs whenever primaryBackendWs/primaryFrontendWs set
  if (primaryBackendWs) {
    const wsAbs = path.join(projectDir, ...primaryBackendWs.split('/'));
    const wsPkg = readPackageJson(wsAbs);
    // D1 scalar language merge
    if (detectLanguage(wsAbs) === 'typescript') language = 'typescript';
    // D2 architecture per-field merge
    const wsArch = detectArchitecture(wsAbs, wsPkg);
    if (wsArch.pattern !== 'unknown') srcStructure.pattern = wsArch.pattern;
    if (wsArch.dirs && wsArch.dirs.length > 0) srcStructure.dirs = wsArch.dirs;
    for (const field of ['hasControllers','hasRoutes','hasModels','hasServices','hasDomain','hasMiddleware','hasFeatures','hasHandlers']) {
      if (wsArch[field] === true) srcStructure[field] = true;
    }
    // D3 backend tests per-field merge
    const wsTests = detectTests(wsAbs, wsPkg);
    backendTests = mergeWorkspaceTests(rootTests, wsTests);
  }

  if (primaryFrontendWs) {
    const wsAbs = path.join(projectDir, ...primaryFrontendWs.split('/'));
    const wsPkg = readPackageJson(wsAbs);
    const wsFrontendAux = detectFrontend(wsAbs, wsPkg);
    for (const field of ['styling','components','state']) {
      if (wsFrontendAux[field] !== null && wsFrontendAux[field] !== undefined) {
        frontend[field] = wsFrontendAux[field];
      }
    }
    const wsTests = detectTests(wsAbs, wsPkg);
    frontendTests = mergeWorkspaceTests(rootTests, wsTests);
  }
}
```

**Tradeoffs**:
- v0.17.2 promotion logic preserved exactly — `workspaceSource` still only set when framework promoted from workspace. No behavior change for the existing v0.17.2 guard (`!backend.framework` / `!frontend.framework`).
- Cost: when root has both backend AND frontend framework detected, we still enumerate all workspaces once to find primaries. For fx's 6 workspaces, this is 6 `readPackageJson` + 6 `detectBackend` + 6 `detectFrontend` calls. Each of those is a `JSON.parse` + dep membership check — trivial cost (<10ms total).
- Loop terminates early once both primaries are found — same as v0.17.2.

**Observability (REVISED v1.2, Codex round-2 IMPORTANT)**: export `primaryBackendWs` and `primaryFrontendWs` as diagnostic fields on `scan.backend` / `scan.frontend`. Keep v0.17.2's `workspaceSource` unchanged (still set only when framework was promoted from workspace). The two fields convey different information:

- `workspaceSource`: "this workspace supplied the framework field because root didn't detect it" — `null` when root has the framework hoisted. Preserves v0.17.2 semantics exactly.
- `primaryWorkspace`: "this is the workspace we used for auxiliary detection (language/tests/architecture/aux)" — always set for monorepos with at least one detectable backend/frontend workspace.

Exporting enables smoke scenarios 76/76b to assert observable outputs (resolving Codex round-2 IMPORTANT finding on internal-state assertions) and improves `--doctor` diagnostics for future debugging.

### Phase 1 — Scanner refactor (**REVISED per round-1**)

**Target**: `lib/scanner.js` only.

1. Move three calls out of the return-object-literal into named variables (lines 92, 97, 98 today):

2. **Replace** the current v0.17.2 block (lines 49-87) with the new single-pass enumeration from D5. v0.17.2's internal promotion logic is preserved byte-equivalent — the only addition is `primaryBackendWs` / `primaryFrontendWs` tracking + subsequent aux detection blocks.

3. Add a private helper `mergeWorkspaceTests(rootTests, wsTests)` per D3. Defined inside scanner.js (not exported) — it's a local helper.

4. Update the return to emit `language`, `srcStructure`, `tests: rootTests`, `backendTests`, `frontendTests`:

```js
return {
  projectName: pkg.name || path.basename(projectDir),
  description: pkg.description || '',
  language,
```

### Phase 2 — Adapt functions + wizard migration (**EXPANDED per Gemini round-1 IMPORTANT**)

**Target**: `lib/init-generator.js` + `lib/init-wizard.js`.

1. `adaptBackendStandards` (init-generator.js:461-462): `scan.tests.framework` → `scan.backendTests.framework`.
2. `adaptFrontendStandards` (init-generator.js:575): `scan.tests.framework` → `scan.frontendTests.framework`.
3. **init-wizard.js summary** (lines 44-48) — **REVISED v1.2, Gemini round-2 IMPORTANT**. v1.1's OR-precedence picked root first, so a monorepo that hoists `jest` at root would print "Tests: jest" in the summary even though `adaptBackendStandards` would correctly write "Vitest" (the workspace framework). To avoid that summary-vs-output contradiction, collect all distinct non-`'none'` frameworks and display them joined:
   ```js
   // Before:
   // if (scanResult.tests.framework !== 'none') { ... }
   // After:
   const fwks = Array.from(new Set([
     scanResult.tests.framework,
     scanResult.backendTests.framework,
     scanResult.frontendTests.framework,
   ].filter((f) => f !== 'none')));
   if (fwks.length > 0) {
     const label = fwks.join(' + ');
     lines.push(`    Tests:         ${label} (${totalFiles} test files)`);
   }
   ```
   A mixed monorepo like fx (vitest in api, jest in web) would summarize as `Tests: vitest + jest (NN test files)` — accurate signal of what the adapt-functions will write, no misleading single-framework claim.

4. **Tracker F001 recommendation** (init-generator.js:236-238, 852) — **REVISED v1.2, Codex round-2 IMPORTANT**. v1.1's disjunction only checked `framework !== 'none'`, missing E2E-only setups. A monorepo with only root `@playwright/test` (E2E suite across workspaces, no unit-test framework) would have `framework === 'none'` on all three but `e2eFramework === 'playwright'` and `testFiles > 0` on root — and still get incorrectly recommended F001. Fix: broaden the gate to ANY test signal:
   ```js
   const hasAnyTestSignal =
     scan.tests.framework !== 'none' ||
     scan.backendTests.framework !== 'none' ||
     scan.frontendTests.framework !== 'none' ||
     scan.tests.e2eFramework !== null ||
     scan.tests.testFiles > 0 ||
     scan.backendTests.testFiles > 0 ||
     scan.frontendTests.testFiles > 0;
   if (!hasAnyTestSignal || <existing coverage checks>) {
     // recommend F001
   }
   ```
   Rationale: avoid false-positive "audit test coverage" recommendations on any project that already has meaningful test infrastructure.
5. `adaptAgentsMd` — **no change**. Empirically verified (Codex + Gemini round-1): does not read `scan.tests` at all (init-generator.js:630-711), builds stack bullets from `scan.backend` / `scan.frontend` fields only.

## Testing invariants (**REVISED per Codex + Gemini round-1**)

Plan v1.0's "byte-identical" and "strict superset" framing was too strong — both models flagged contradictions (Codex #2 + #3). Invariants restated per-field:

**1. Existing-fields contract for single-package projects** (`!isMonorepo`):
- All existing scan fields (`language`, `backend`, `frontend`, `srcStructure`, `tests`, etc.) produce values identical to v0.17.2.
- New fields `backendTests` and `frontendTests` are reference-equal to `tests` (`scan.backendTests === scan.tests === scan.frontendTests`).
- Object shape changes: new fields added to the return object. This IS an API shape change. Scenario 63c continues to assert specific fields (not deep snapshot), so it holds. Scenario 79 documents the reference-equality contract for the new fields.

**2. Per-field monotonicity for monorepos** (replacing "strict superset"):
- `language`: `'javascript'` → `'typescript'` only (never demoted); preserves root detection if workspace is also JS.
- `srcStructure.pattern`: `'unknown'` → known pattern only (never demoted from real pattern to unknown).
- `srcStructure.dirs`: replaced when workspace has non-empty array (root's empty `[]` carries no info).
- `srcStructure.hasX`: **OR-merge** — if root or workspace saw it, preserved as `true`. Never demoted.
- `frontend.styling/components/state`: `null` → string only. Root detection preserved if workspace didn't find a signal.
- `backend`/`frontend` core fields (framework/orm/db/etc.): **unchanged from v0.17.2 semantics** — workspace promotion only if root lacked framework. Single-pass loop preserves the `!backend.framework` / `!frontend.framework` guards exactly.
- `backendTests` / `frontendTests` per-field merged from `mergeWorkspaceTests(rootTests, wsTests)`: unit framework from workspace if detected, E2E from whichever set it, counts from whichever saw more.
- `tests` (root-level legacy): untouched by aux detection. Remains equal to `detectTests(projectDir, pkg)`.

**3. Enumeration determinism**: workspace enumeration uses `enumerateWorkspaces` (unchanged from v0.17.1). Deterministic order, dedupe by POSIX path. First-match-wins for `primaryBackendWs` / `primaryFrontendWs` within the new single-pass loop. Same ordering guarantees as v0.17.2.

**4. Cost bound**: single-pass enumeration reads each workspace's package.json at most once, calls `detectBackend` + `detectFrontend` at most once each per workspace. For fx's 6 workspaces: ≤ 6 readPackageJson + ≤ 6 detectBackend + ≤ 6 detectFrontend calls → <10ms. Plus aux detection on ≤ 2 primary workspaces (language/arch/tests on one, frontend aux/tests on another).
