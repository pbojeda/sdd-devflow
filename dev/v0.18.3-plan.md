## 2. Empirical evidence

### 2.1 B8 — P6 false positive on deferred-AC tickets

Source: ROADMAP "Known follow-ups" item 8 (carried since 2026-04-28).
Recipe (current, `template/.claude/commands/audit-merge.md:107-108`):

```bash
grep -oE 'all [0-9]+ marked|AC: [0-9]+/[0-9]+' <ticket> | head -1 | grep -oE '[0-9]+' | head -1
```

When ticket MCE row says `AC: 11/13 done`, the `head -1` extraction returns `11` (CLAIMED), then `ACTUAL` is computed as total checkbox count = 13. CLAIMED (11) ≠ ACTUAL (13) → spurious P6 IMPORTANT flag on a ticket with legitimately-deferred ACs (e.g. F-H10-FU: 11 [x] + 2 [ ] = 13 total, claim is accurate).

**Empirical hits**: F-H10-FU (2026-04-28 audit), F-MULTITURN-001 (2026-05-07 audit indirectly — would have hit if `AC: 17/17` had been written as `AC: 15/17 done + 2 deferred`).

**Fix candidate**: when `AC: X/Y` form matches, extract BOTH numbers and compare `Y` (total claimed) to checkbox total. If `Y == total`, PASS (accuracy of claim). Only flag when `Y ≠ total`. Marked-count drift (`X` vs `[x]` count) becomes a separate sub-check.

### 2.2 B9 — P5 sed parenthetical / em-dash post-Done

Source: ROADMAP item 8 + fx F-WEB-MENU-VISION-001 audit 2026-05-06.
Recipe (current after v0.18.1 B6 harden):

```bash
status=$(grep -E "^\\*\\*Status:\\*\\*" <ticket> | head -1 \
  | sed -E 's/^\*\*Status:\*\* *\*\*?//; s/\*\*//g; s/ *\|.*//')
```

For `**Status:** Done (code merged 2026-04-22; prod DB migration executed 2026-04-23)` → `status="Done (code merged 2026-04-22; prod DB migration executed 2026-04-23)"`. Subsequent `[ "$status" = "Done" ]` test fails → ticket counted as frozen.

**Empirical hits (fx 2026-05-06 re-audit)**: BUG-PROD-009, F114, F-UX-B, BUG-PROD-008-FU1 — ~10 tickets feeding FROZEN_COUNT=52 noise.

**Fix candidate (preferred)**: extra sed pass `sed -E 's/[[:space:]]+(\(.*\)|—.*|–.*|-.*).*$//'` before the existing `s/ *\|.*//` strip. POSIX-compliant. Test fixtures must cover:
- `Done` (bare)
- `Done (parenthetical)`
- `Done — em-dash`
- `Done – en-dash`
- `Done - hyphen`
- `Done | pipe` (pre-existing case)
- `In Progress` (negative — must NOT collapse to `In`)

### 2.3 P13 — key_facts delta vs ticket atom-count mismatch (NEW)

Source: 2026-05-08 audit of fx F-MULTITURN-001 (key_facts log row claimed `+7 atoms` but ticket detail listed 8 atom IDs).

**Pattern**: when a ticket's Completion Log or MCE references `key_facts.md` and quantifies a delta (e.g. `+N atoms`, `+M aliases`, `+K dishes`), the corresponding `key_facts.md` row should have the same N/M/K.

**Recipe sketch** (R1 M1 fix — `while IFS= read -r` to avoid whitespace-splitting of multi-word deltas; anchor on FEATURE_ID to avoid false-pass on unrelated identical deltas elsewhere in key_facts.md):
```bash
# 1. Extract ticket-side delta claims (one per line; multi-word safe)
FEATURE_ID=$(basename "$TICKET" .md | sed -E 's/-[a-z].*//')
TICKET_DELTAS=$(grep -oE '\+[0-9]+ (atoms?|aliases?|dishes?|entries|rows)' "$TICKET" | sort -u)
# 2. Extract key_facts.md rows mentioning this feature
KEY_FACTS_BLOCK=$(grep -A 3 -F "$FEATURE_ID" docs/project_notes/key_facts.md 2>/dev/null || true)
# 3. For each delta claim, require it to appear in the feature's key_facts block (not anywhere in the file)
while IFS= read -r claim; do
  [ -z "$claim" ] && continue
  if ! echo "$KEY_FACTS_BLOCK" | grep -qF "$claim"; then
    flag "P13 drift: ticket claims '$claim' but key_facts.md block for $FEATURE_ID lacks the same delta"
  fi
done <<< "$TICKET_DELTAS"
```

**Caveats**:
- Plural/singular variance (`atom` vs `atoms`) → regex already covers via `s?`.
- Multi-line table rows → `grep -A 3` after the feature-ID anchor (tunable).
- Bilingual: Spanish-only key_facts (`+8 átomos`) needs separate path — **deferred to v0.19.x** per R1 consensus.

**Severity**: IMPORTANT (data integrity claim).

### 2.4 P14 — MCE Action 1 stale post-merge (NEW)

Source: 2026-05-08 audit of fx F-CATALOG-COV-001 (MCE row 1 said `Workflow Steps 0-5 [x], Step 6 [ ] (this merge)` — `Step 6 [ ]` should have flipped post-merge per F-MULTITURN-001 precedent).

**Pattern**: ticket Status = `Done` (post-merge) AND MCE table contains `Step 6 [ ]` or `Step 6 [-]` or `(this merge)` phrasing → stale.

**Recipe sketch**:
```bash
# Only run when ticket Status normalizes to "Done" (B9 fix must land first for clean extraction)
if [ "$status" = "Done" ]; then
  if grep -E '(Step 6 \[ \]|Step 6 \[-\]|\(this merge\))' <ticket-mce-section> >/dev/null; then
    echo "P14 NIT: MCE Action 1 row stale post-merge — flip Step 6 checkbox + remove (this merge) qualifier"
  fi
fi
```

**Section-scoping**: only check within the MCE table fence (between `## Merge Checklist Evidence` and next `## ` header).

**Severity**: NIT (cosmetic / doc hygiene; doesn't block merge but accumulates over time).

### 2.5 P15 — AC `post-deploy` keyword admitted as [x] without Completion Log evidence (NEW)

Source: 2026-05-08 audit of fx F-CATALOG-COV-001 AC-NEW-qa-battery (created in R6 as a production-parity gate; marked `[x]` at merge but no Completion Log row with QA battery results until external audit forced it — resolved at commit `81eea5c` PR #263).

**Pattern**: AC line text contains `post-deploy` (or `post-merge` or `production parity` or `prod verification`) AND checkbox is `[x]` AND no Completion Log row anchors that AC ID with dated evidence.

**Recipe sketch** (R1 M1 fix — `while IFS= read -r` to keep each full AC line intact; whitespace-splitting would corrupt the `[x]` / AC-ID extraction):
```bash
# 1. Scope Completion Log section once
COMPLETION=$(awk '/^## Completion Log/,/^## Merge Checklist/' "$TICKET")
# 2. Find AC lines with post-deploy keywords (one full line per record)
AC_LINES=$(grep -nE '^[[:space:]]*-[[:space:]]*\[[ x]\][[:space:]]+AC-[A-Z0-9_-]+' "$TICKET" \
  | grep -iE 'post-deploy|post-merge|production parity|prod verification|on dev API|on prod')
# 3. For each [x] match, search Completion Log for AC-ID + dated row
while IFS= read -r line; do
  [ -z "$line" ] && continue
  echo "$line" | grep -q '\[x\]' || continue
  ac_id=$(echo "$line" | grep -oE 'AC-[A-Z0-9_-]+' | head -1)
  [ -z "$ac_id" ] && continue
  if ! echo "$COMPLETION" | grep -qE "^\|[[:space:]]*[0-9]{4}-[0-9]{2}-[0-9]{2}[[:space:]]*\|.*$ac_id"; then
    flag "P15 drift: $ac_id marked [x] with post-deploy semantics but no dated Completion Log entry"
  fi
done <<< "$AC_LINES"
```

**Severity**: IMPORTANT — these ACs are explicit production gates; silent-PASS defeats their purpose.

**Open question for R1**: should this be a new pattern (P15) or a refinement of P2 (Aspirational MCE rows)? P2 catches "will" / "to be" wording; P15 catches the inverse — admitted as done without evidence. They are complementary, recommend separate patterns.

### 2.6 C1 — P1 multi-workspace test-count grab

Source: ROADMAP item 9 (2026-05-06 fx audit).

Current P1 recipe extracts FIRST `[0-9]+/[0-9]+` from any line containing test keywords. In fx (api 4272, web 487, bot 738/3, shared 598, scraper 1221, landing 232), the FIRST ratio depends on whichever workspace ran first.

**Fix candidate**: extract ALL `[0-9]+/[0-9]+` ratios from terminal evidence AND verify each appears in the ticket's `Tests:` line OR Completion Log. If ticket claims a global number (e.g. `4094/4094`), verify it matches `--reporter=verbose` SUM across workspaces.

**Severity bump**: today P1 is IMPORTANT but shallow. Post-fix it remains IMPORTANT with deeper signal.

### 2.7 C2 — P11 NIT extension / new P16

Source: ROADMAP item 9 (fx F-WEB-MENU-VISION-001 had no `## Features — *` table row; agent claimed "by design standalone").

**Pattern**: ticket Status = `Ready for Merge` OR `Done` AND no row in any `## Features — *` table mentions the ticket ID.

**Recipe sketch**:
```bash
ticket_id=$(grep -m1 -oE '^# (F-[A-Z0-9_-]+|BUG-[A-Z0-9_-]+)' <ticket> | sed 's/^# //')
if ! grep -E "^\| .* $ticket_id" docs/project_notes/product-tracker.md >/dev/null; then
  if [ "$status" = "Done" ] || [ "$status" = "Ready for Merge" ]; then
    echo "P16 NIT: $ticket_id not in any Features table"
  fi
fi
```

**Decision**: implement as **new P16** (independent recipe), not as P11 mutation. Keeps P11 semantics tight (status-mismatch only).

## 7. PM Orchestrator + L5 audit framework — DEFERRED to v0.19.0 plan

User-flagged 2026-05-08: PM Orchestrator + L5 autonomous mode is a primary library differentiator that has shipped (v0.16.0, refined through v0.16.6) but lacks dedicated audit coverage. Current state:

- `/audit-merge` audits per-ticket post-merge (structural + drift).
- `/audit-feature` audits in-flight feature scope.
- **No** `/audit-pm-session` or analogue audits the **session-level** invariants for an L5 autonomous run (12 guardrails, baseline check, mandatory compact, intra-batch dependency, check existing patterns, etc.).

**Empirical motivation**: two PM sessions shipped (pm-c3a + pm-uf3 = 6/6 APPROVED) and one more in-flight (pm-h6plus). External audits caught real issues in individual tickets but none at the session-level orchestration layer. As more L5 sessions ship, drift in session-state (`.lock` file lifecycle, compact hook PM-awareness, baseline drift across feature batches) becomes a class of bug that no current recipe catches.

**Proposed scope for v0.19.0** (separate plan to draft):

1. **`/audit-pm-session`** slash command — operates over a pm-session.md + lock file pair. Checks:
   - Lock file lifecycle (created at session start, deleted at end OR session marked `complete`)
   - Baseline SHA recorded matches git's state at session start (within ±N commits)
   - Each feature in batch has its own ticket + each ticket has Step 6 housekeeping or is explicitly deferred
   - Compact-hook PM-awareness invariant (compact happens between features, not mid-feature)
   - Intra-batch dependency check (no feature B references feature A's commits that aren't merged yet)
2. **L5 autonomy invariants doctor check** (#16) — when `--doctor` runs in a project that has ever had an L5 session, validate `.claude/skills/pm-orchestrator/` is present + intact + matches expected hashes.
3. **PM session retrospective recipe** — analogue to `/audit-merge` but session-scoped: drift between pm-session.md goal statement and actual shipped features.

Defer rationale:
- v0.18.3 is a patch (drift hardening). PM/L5 audit is a new capability (minor version).
- Requires its own R1 cycle, fixtures, and empirical validation against pm-h6plus + future sessions.
- Spec needs co-design with user on which guardrails matter most (12 candidates exist; auditing all 12 is overkill).

**Action**: open `dev/v0.19.0-pm-l5-audit-plan.md` after v0.18.3 ships, NOT before.
