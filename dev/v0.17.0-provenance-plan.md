# Plan: sdd-devflow-pb v0.17.0 — Provenance tracking for accurate cross-version smart-diff

**Status**: v1.1 — cross-model reviewed (Codex CLI + Gemini CLI), consensus findings incorporated
**Version target**: 0.17.0 (minor bump — new feature, no breaking CLI changes)
**Priority**: HIGH — closes Codex P1 finding from v0.16.10 cross-model review

---

## Context

v0.16.10 shipped smart-diff protection for template agents and `AGENTS.md` (see `CHANGELOG.md § 0.16.10`). The core mechanism: compare the user's on-disk file against `adaptAgentContentString(rawTemplate, file, projectType)` — match → replace, mismatch → preserve and save `<path>.new`.

This works for same-version upgrades but has a known limitation for **cross-version upgrades**: when a template agent or `AGENTS.md` changes between releases, a pristine user file won't match the new adapted target, so the smart-diff flags it as "customized" and preserves it. The user has to re-run with `--force-template` or manually merge every file. Every future version bump that touches any template file triggers this friction.

Codex raised this as **P1** in post-implementation review. The same concern was raised as **H-severity** in the pre-implementation plan review and explicitly deferred to v0.17.0 with provenance tracking. v0.16.10 accepted conservative preserve semantics + `--force-template` escape hatch + CHANGELOG documentation as the trade-off.

**Additional cross-path drift discovered during v0.16.10 implementation**: `init-generator.js`'s `adaptCopiedFiles` applies stack-specific adaptations (Zod → validation, `shared/src/schemas/` cleanup, Prisma → detected ORM, DDD → layered) to 9 template files. These adaptations are **not** mirrored in `upgrade-generator.js`, so init-adapted files look "customized" to the v0.16.10 smart-diff on upgrade. Smoke scenario 45 was reworked as a unit test to avoid this case because full init→upgrade convergence requires unifying the adaptation pipeline.

**v0.17.0 must solve BOTH drifts simultaneously**: cross-version (Codex P1) AND cross-path init↔upgrade (smoke 45 drift). The only clean solution is to **unify** the stack-adaptation pipeline (shared module, callable from init and upgrade) AND introduce content-addressable provenance (`.sdd-meta.json` with hashes).

## Goals

1. **Eliminate cross-version false-positive preserve warnings** on cross-version upgrades for files the user hasn't touched. Hash answers "did the user edit since last tool-write" precisely.
2. **Unify init + upgrade adaptation pipelines** via a shared `lib/stack-adaptations.js` module so the same `(content, scan, config)` input produces the same output regardless of caller.
3. **Preserve v0.16.10's foodXPlorer-regression fix**: smart-diff still protects customized files from silent clobber. The fix for Codex M1 is critical here — preserved files MUST NOT get new hashes.
4. **Graceful fallback** for pre-v0.17.0 projects. First upgrade from v0.16.10 must not crash; it should use an in-memory stack-adapted comparison target (not the bare core target) and write `.sdd-meta.json` for subsequent upgrades.
5. **Zero runtime/dev deps**.
6. **No breaking CLI changes**. `--force-template` from v0.16.10 remains.

---

## Non-goals (v0.17.0)

- Hash-based smart-diff for `SKILL.md`, `ticket-template.md`, `documentation-standards.mdc` — these continue with wholesale-recopy + stack-adaptation on every upgrade (v0.16.10 behavior). First-class smart-diff for them is **v0.17.x**.
- Hash-based smart-diff for `ai-specs/specs/*.mdc` standards — existing `isStandardModified` content-compare stays. Refactor deferred.
- Rich metadata beyond hashes (install timestamps, per-version history, rule-set version). v0.17.0 stores only `schemaVersion` + `hashes`. v0.17.x can add fields non-breaking.
- Unifying `generator.js`'s `adaptAgentsForStack` (wizard-driven) with `init-generator.js`'s `adaptCopiedFiles` (scan-driven). They use different inputs and solve different cases. The shared `lib/stack-adaptations.js` module is extracted from `adaptCopiedFiles` only. `generator.js` continues with its lighter pipeline; first upgrade on a create-scaffolded project re-adapts from scanner (see M2 resolution § Scanner-as-source-of-truth).
- First-class integration of doctor check #15 with `--upgrade --diff` preview mode. Can come later.

## Design

### `.sdd-meta.json` schema (v1)

Written to the project root alongside `.sdd-version`. Ignored by git via `.gitignore`.

```json
{
  "schemaVersion": 1,
  "hashes": {
    ".claude/agents/backend-planner.md": "sha256:abc123...",
    ".claude/agents/backend-developer.md": "sha256:def456...",
    ".claude/agents/spec-creator.md": "sha256:...",
    ".claude/agents/code-review-specialist.md": "sha256:...",
    ".claude/agents/database-architect.md": "sha256:...",
    ".claude/agents/frontend-planner.md": "sha256:...",
    ".claude/agents/frontend-developer.md": "sha256:...",
    ".claude/agents/production-code-validator.md": "sha256:...",
    ".claude/agents/qa-engineer.md": "sha256:...",
    ".claude/agents/ui-ux-designer.md": "sha256:...",
    ".gemini/agents/backend-planner.md": "sha256:...",
    "AGENTS.md": "sha256:..."
  }
}
```

- `schemaVersion: 1` — bump on breaking format changes. Additions within v1 MUST be backward-compatible (old readers ignore unknown fields). Readers with `schemaVersion` absent treat as v1.
- `hashes` — map of `POSIX-normalized relative path → "sha256:<hex>"`. Only files that participate in hash-based smart-diff: template agents (`.claude/agents/*`, `.gemini/agents/*`) + `AGENTS.md`. That's **at most 21 entries** for a fullstack `both` project, fewer for single-stack or single-tool variants.
- `SKILL.md`, `ticket-template.md`, `documentation-standards.mdc` are **not** in this map. They continue with wholesale-recopy + stack-adaptation on every upgrade (v0.16.10 behavior).
- `ai-specs/specs/*.mdc` standards are **not** in this map. Existing `isStandardModified` handles them.
- Hash format `sha256:<hex>` — prefix is mandatory so v0.17.x can introduce other algorithms non-breaking.

### Hash algorithm and normalization

```js
const { createHash } = require('node:crypto');

function normalizeForCompare(text) {
  // v0.17.0: only strip CR/CRLF. Do NOT strip trailing whitespace —
  // that would destroy markdown hard-breaks (two trailing spaces = <br>).
  return text.replace(/\r\n/g, '\n').replace(/\r/g, '\n');
}

function computeHash(content) {
  return 'sha256:' + createHash('sha256').update(normalizeForCompare(content), 'utf8').digest('hex');
}
```

**Trade-off documented**: editors that strip trailing whitespace on save (e.g. VSCode `"files.trimTrailingWhitespace": true`) will produce a hash mismatch even without semantic edits. This is a conservative false positive → preserve warning → user runs `--force-template` to re-accept. Acceptable; the alternative (silent wipe of user's markdown hard breaks) is worse.

### POSIX path normalization (Windows)

```js
const path = require('node:path');

function toPosix(relativePath) {
  return relativePath.split(path.sep).join('/');
}
```

Applied at every read/write of `meta.hashes[key]`. Canonical lookup form: POSIX.

### In-memory stack adaptations (Gemini M1 fix)

`applyStackAdaptationsToContent` is implemented by factoring the existing file-based rule application into a pure function:

```js
function applyStackAdaptationsToContent(content, posixRelativePath, scan, config) {
  const rulesForFile = computeRulesFor(posixRelativePath, scan, config);
  let result = content;
  for (const [search, replace] of rulesForFile) {
    if (search instanceof RegExp) {
      result = result.replace(search, replace);
    } else {
      result = result.split(search).join(replace);
    }
  }
  return result;
}
```

Where `computeRulesFor(posixRelativePath, scan, config)` returns the list of rules that would apply to this specific file given the project state. The rules are factored out of the current `adaptCopiedFiles` imperative body into a data-driven table keyed by file path.

The file-based `applyStackAdaptations` calls `applyStackAdaptationsToContent` internally:

```js
function applyStackAdaptations(dest, scan, config, replaceInFileFn, allowlist = null) {
  const touched = [];
  for (const posixPath of candidateFilesFor(scan, config)) {
    if (allowlist && !allowlist.has(posixPath)) continue;
    const absPath = path.join(dest, ...posixPath.split('/'));
    if (!fs.existsSync(absPath)) continue;
    const before = fs.readFileSync(absPath, 'utf8');
    const after = applyStackAdaptationsToContent(before, posixPath, scan, config);
    if (before !== after) {
      fs.writeFileSync(absPath, after, 'utf8');
    }
    touched.push(posixPath);
  }
  return touched;
}
```

Note: `replaceInFileFn` is no longer needed as a parameter since the transformation is now data-driven. The previous `regexReplaceInFile` vs `replaceInFile` split existed because init used `.replaceAll` semantics while generator used `.replace` first-occurrence. The new design uses explicit `split().join()` for string rules and `.replace()` for regex rules, matching `adaptAgentContentString`'s behavior in `lib/adapt-agents.js`. This means **rules that previously relied on first-occurrence-only semantics would change behavior**. Audit step during implementation: check every rule in `adaptCopiedFiles` for dependence on first-occurrence.

~100 lines. Exports:

### Smart-diff decision tree (revised — Codex M1 fix)

**Key invariant**: `meta.hashes[file]` is a canonical marker that says "the last time this tool wrote this file, the content hashed to X". It is **only updated** when the tool actually writes canonical content to this file in the current run. Preserved files leave the entry untouched.

### Install-time hash write (generator.js + init-generator.js)

- **`lib/init-generator.js`** (`generateInit`): after all adaptations run (including `applyStackAdaptations`), compute hashes for all files in the `TEMPLATE_AGENTS + AGENTS.md` set that exist on disk for the current `(aiTools, projectType)`. Write `.sdd-meta.json`. Single-stack-pruned files are absent from the map.
- **`lib/generator.js`** (`generate`): same, but NOTE: generator.js runs `adaptAgentsForStack` (wizard-driven) instead of `applyStackAdaptations` (scanner-driven). The hash captures whatever generator.js's pipeline produced.

If the scanner detects the wizard-predicted stack (happy path), the re-adapted output matches the install-time output → hash still reflects the current on-disk content → replacement is a no-op (same content). If the scanner detects something else, the re-adaptation reflects reality. The wizard's `bPreset` is an initial guess; the scanner is the authoritative source post-install. Documented in § Scanner-as-source-of-truth.

### Scanner-as-source-of-truth (Codex M2 design note)

v0.17.0 intentionally does NOT persist wizard config (`bPreset.orm`, `bPreset.db`) in `.sdd-meta.json`. The reasoning:

1. The wizard captures the user's **intent** at scaffold time. Intent can diverge from reality (user predicts Prisma but installs Mongoose).
2. The scanner captures **ground truth**: what's actually in `package.json` and source directories post-install.
3. Upgrades should follow reality, not stale predictions. When the user upgrades SDD, they want adaptations that match their current stack, not their 6-month-old prediction.

Concretely: a project scaffolded with `create-sdd-project --yes` gets `bPreset.orm = 'Prisma'` (default) → `generator.js` writes "Prisma ORM" references. The user then installs Mongoose instead. First upgrade: scanner detects Mongoose, `applyStackAdaptations` rewrites references to "Mongoose". This is CORRECT — the file now matches the project's actual stack.

The subtle case: hash-match says "pristine" (user hasn't edited), so upgrade replaces with new adapted content. If the wizard's prediction matched reality, the new adapted content is byte-identical → no visible change. If the prediction didn't match reality, the new adapted content is different → user sees the scanner's read of their actual stack. Either way, the outcome reflects current reality.

Where this would be a regression: if a user intentionally customized their agent files to use the WRONG stack (e.g. pretending their Mongoose project is Prisma), the hash-match would overwrite that. But this is an adversarial case — customization of this form would change the file hash at the moment the user edited it, so the hash would mismatch → preserve. The scenario only bites if the user edits OUTSIDE the tracked content (e.g., adds a comment elsewhere), which doesn't change the stack adaptation substrings.

Smoke Scenario 58 (new — add to the v0.17.0 test plan) will verify this:

- Scaffold via `create-sdd-project` with `backendPreset = Prisma`
- Modify `package.json` + `node_modules` simulation to detect `Mongoose` (or simpler: manually edit `backend-developer.md` to replace "Prisma" with "Mongoose")
- Run `--upgrade --force --yes`
- Assert: the replacement applies scanner-driven adaptations, not wizard-driven

### Allowlist semantics

`applyStackAdaptations(dest, scan, config, _, allowlist)` receives a Set of POSIX paths that it's allowed to touch. The purpose is to prevent stack adaptations from running on **preserved** user-edited files.

- **Install path** (init): allowlist is `null` — all candidate files are touched. Same behavior as v0.16.10.
- **Upgrade path**: allowlist is `filesToAdapt` — ONLY the files that were replaced by smart-diff this run. Preserved files are not in the set. Files outside the smart-diff loop (SKILL.md, ticket-template.md, documentation-standards.mdc) are ALWAYS in the set because upgrade wholesale-recopies them BEFORE the smart-diff loop, so they're always in the "replaced" state.

Concretely, in `upgrade-generator.js`:

```js
// Pseudocode

                    '.claude/skills/development-workflow/references/ticket-template.md',
                    /* etc */ ]) {
  copyFromTemplate(path);
  filesToAdapt.add(toPosix(path));
}

// Step B: smart-diff template agents + AGENTS.md, add replaced ones to filesToAdapt
// (see decision tree above)

// Step C: apply stack adaptations to the allowlist only
applyStackAdaptations(dest, scan, config, replaceInFileFn, filesToAdapt);

// Step D: re-hash smart-diff files that were replaced
for (const posixPath of filesToAdapt) {
  if (SMART_DIFF_TRACKED_PATHS.has(posixPath)) {
    newHashes[posixPath] = hashFileOnDisk(path.join(dest, ...posixPath.split('/')));
  }
}

// Step E: prune expected-absent
newHashes = pruneExpectedAbsent(newHashes, aiTools, projectType);

// Step F: write
writeMeta(dest, newHashes);
```

### Expected-presence pruning (Gemini M3 fix)

```js
function pruneExpectedAbsent(hashes, aiTools, projectType) {
  const expected = expectedSmartDiffTrackedPaths(aiTools, projectType);
  // expected is the full set of POSIX paths this (aiTools, projectType) should have.
  // Returns a new map with entries for paths NOT in `expected` removed.
  const pruned = {};
  for (const [k, v] of Object.entries(hashes)) {
    if (expected.has(k)) pruned[k] = v;
  }
  return pruned;
}

function expectedSmartDiffTrackedPaths(aiTools, projectType) {
  const paths = new Set();
  const toolDirs = [];
  if (aiTools !== 'gemini') toolDirs.push('.claude');
  if (aiTools !== 'claude') toolDirs.push('.gemini');
  const agents = TEMPLATE_AGENTS.filter((a) => {
    if (projectType === 'backend' && FRONTEND_AGENTS.includes(a)) return false;
    if (projectType === 'frontend' && BACKEND_AGENTS.includes(a)) return false;
    return true;
  });
  for (const dir of toolDirs) {
    for (const agent of agents) paths.add(`${dir}/agents/${agent}`);
  }
  paths.add('AGENTS.md');
  return paths;
}
```

A user who temporarily deletes `backend-planner.md` keeps its hash in `meta.hashes`. Next upgrade: file doesn't exist → first branch of decision tree (writeFileSync adaptedCoreTarget + add to filesToAdapt) → hash is recomputed to the newly-written content's hash. No information loss.

### Doctor check #15 (Codex M3 fix)

Validates `.sdd-meta.json` structural integrity **only**:

- File exists → try to parse as JSON
- `schemaVersion` is absent OR a positive integer ≤ `CURRENT_SCHEMA_VERSION`
- `hashes` is an object (not null, not array)
- Every key in `hashes` is a string matching a sensible path shape (POSIX, no `..`, no absolute)
- Every value in `hashes` is a string matching `^sha256:[0-9a-f]{64}$`
- Every key that's expected for this (aiTools, projectType) either exists in `hashes` or does not exist on disk (pruned/absent is OK)
- Every key in `hashes` that's NOT expected for this (aiTools, projectType) is flagged as "orphan entry"

**Not** validated by doctor #15:
- Whether hashes match current on-disk content. **Reason**: mismatch is the expected result of user customization. Reporting it would generate permanent noise and bury real integrity issues.

Severity:
- File missing → PASS with message "no provenance tracking (pre-v0.17.0 project or new)"
- File present and all checks pass → PASS
- Parse error, wrong shape, invalid schema version → WARN (not FAIL — upgrade still has a working fallback path)
- Orphan entries → WARN (non-fatal, upgrade will prune next run)

### Fallback guarantees (scenario matrix)

| Scenario | Has `.sdd-meta.json` on entry | On exit, `.sdd-meta.json` has hashes? |
|---|---|---|
| A. Create (generator.js) fresh scaffold | N/A (new project) | Yes, computed from final install-time content |
| B. --init on existing project | N/A (new install) | Yes, computed after `applyStackAdaptations` |
| C. Upgrade on pre-v0.17.0 (v0.16.x) project — fallback path | No | Yes, computed for every file that was **replaced** by the fallback path. Preserved files get **no** hash entry. |
| D. Upgrade on v0.17.0+ project — hash path happy case | Yes | Yes, all tracked files' hashes updated (replaced or pristine). |
| E. Upgrade on v0.17.0+ with user customization | Yes | Yes, **but** preserved files' old hash entries **do not change**. Only replaced files' hashes are updated. |
| F. Corrupted `.sdd-meta.json` on entry | Invalid | Yes, rewritten with fresh hashes (fallback path was used). |
| G. User manually deletes `.sdd-meta.json` | No | Yes, rewritten (same as F). |

**Scenario E is the critical one for Codex M1**: a user customizes `backend-planner.md`. Upgrade detects hash mismatch → preserve → `modifiedAgentsResults` entry → `.new` backup. The stored hash is LEFT UNCHANGED. Next upgrade: user hasn't touched the file again (content still has their customization) → currentHash still mismatches stored hash → still preserved. The silent-clobber vector is closed.

**Sub-case**: user customizes, then reverts to the exact tool-canonical content. Next upgrade: currentHash matches stored hash → replaced → adaptation applied → new hash written. Correct.

## Risks (updated)

### Risk 1 (addressed): Codex M1 — preserved files get hashed → silent clobber
**Resolution**: invariant "hash only when canonical output was written this run". Scenario 54 as regression guard.
**Residual risk**: LOW if the invariant is enforced correctly in the upgrade code path. Code review must verify no code path updates `newHashes[posixPath]` in a preserve branch.

### Risk 2 (addressed): Codex M1 — SKILL.md / ticket-template.md / documentation-standards.mdc tracking gap
**Resolution**: narrowed v0.17.0 smart-diff to template agents + AGENTS.md only. Other stack-adapted files continue with wholesale-recopy + stack-adaptation on every upgrade, as in v0.16.10. First-class smart-diff for them is a v0.17.1 candidate.
**Residual risk**: LOW. These files were never protected by v0.16.10 either; v0.17.0 does not regress them.

### Risk 3 (addressed): Gemini M1 — fallback path false positives for --init projects
**Resolution**: `applyStackAdaptationsToContent` applied to the comparison target in the fallback path. Scenario 55 as regression guard.
**Residual risk**: LOW if `applyStackAdaptationsToContent` produces byte-identical output to the file-based `applyStackAdaptations` for the same input. Scenario 56 (idempotency) + a parity test covers this.

### Risk 4 (addressed): Codex M2 — create→upgrade adaptation divergence
**Resolution**: documented as intentional "scanner-as-source-of-truth" semantics. Scenario 58 proves the expected behavior.
**Residual risk**: LOW. The user's wizard-config is explicitly not persisted; upgrades follow reality. If this surprises users, it can be revisited in v0.17.x by persisting `bPreset` in `.sdd-meta.json`.

### Risk 5 (addressed): Gemini M2 — Windows path normalization
**Resolution**: `toPosix` helper applied consistently at every read/write of `meta.hashes` keys. Documented in § POSIX path normalization.
**Residual risk**: LOW if applied consistently. Code review + smoke scenario using mixed separators would catch misses.

### Risk 6 (addressed): Gemini M2 — markdown hard-breaks destroyed by normalization
**Resolution**: `normalizeForCompare` only strips `\r`/`\r\n`. Trailing whitespace preserved. Trade-off documented: VSCode "trim trailing whitespace on save" produces a conservative false-positive; `--force-template` is the escape hatch.
**Residual risk**: LOW. The trade-off is biased toward the conservative direction (false-positive preserve is better than silent wipe).

### Risk 7 (addressed): Codex M3 — doctor #15 hash-mismatch noise
**Resolution**: doctor #15 focuses on structural integrity only. Hash-mismatch reporting stays in `--upgrade` summary output.
**Residual risk**: LOW.

### Risk 8 (addressed): Gemini M3 — hash map pruning loses entries for user-deleted files
**Resolution**: prune based on "expected presence" per (aiTools, projectType), not on-disk presence. User-deleted files keep their hash; upgrade recreates the file on next run.
**Residual risk**: LOW.

### Risk 9 (new): `computeRulesFor` factoring introduces regressions in `adaptCopiedFiles`
**Description**: the current `adaptCopiedFiles` is imperative. Extracting the rules into a data-driven table requires careful factoring. Any loss of fidelity would change the hash of install-time content for some combinations of `(scan, config)`.
**Mitigation**: parity test — run the new extracted `applyStackAdaptations` on a baseline project and compare byte-for-byte against the current inline function. Add this as a pre-implementation verification step.
**Residual risk**: MEDIUM — factoring gotchas are common. The parity test is the primary line of defense.

### Risk 10 (new): `.replace` → `split().join()` semantic change for some string rules
**Description**: the current `regexReplaceInFile` and similar helpers use `content.replace(search, replace)` for string rules — replaces only the **first** occurrence. The new `applyStackAdaptationsToContent` uses `content.split(search).join(replace)` — replaces **all** occurrences. Any rule that relied on first-occurrence semantics would now behave differently.
**Mitigation**: during implementation, audit every rule in `adaptCopiedFiles`. Look for source strings that appear more than once in any template file. If found, the rule either depended on the first-occurrence quirk (in which case the semantic change is a latent bug we're now exposing) or was fine with both (in which case no change). Document findings in the implementation commit message.
**Residual risk**: MEDIUM — requires manual inspection. The rules touch 9 files with roughly 30-40 total rules.

### Risk 11 (new): `applyStackAdaptationsToContent` + file-based `applyStackAdaptations` output divergence
**Description**: the two entry points must produce identical output for the same input. If the in-memory variant has a bug, the fallback comparison target would differ from the install-time file, causing false positives.
**Mitigation**: the file-based variant is a thin wrapper around the in-memory variant (reads, calls in-memory, writes back if changed). Single source of truth.
**Residual risk**: LOW if the wrapper is thin.

### Risk 12 (new): `.sdd-meta.json` race conditions
**Description**: if two upgrade processes run concurrently (unlikely but possible in CI), the last writer wins.
**Mitigation**: not addressed in v0.17.0. Users running concurrent upgrades is an adversarial case.
**Residual risk**: LOW. Document as "do not run concurrent upgrades".
