## 5. Bug-by-bug fix specification

### Fix B1 — P1 regex broaden (`audit-merge.md:58-59`)

**Current:**
```bash
PR_TESTS=$(echo "$PR_BODY" | grep -iE "(npm test|tests?.*(pass|green))" | grep -oE "[0-9]+/[0-9]+" | head -1)
TICKET_TESTS=$(grep -iE "(npm test|tests?.*(pass|green))" "$TICKET" | grep -oE "[0-9]+/[0-9]+" | tail -1)
```

**Problem:** PR body wordings like `**Tests**: 4094 → 4110 (+16)` don't contain `npm test`, `pass`, or `green` keywords.

**Fix:** add `[Tt]ests?` keyword as alternation:
```bash
PR_TESTS=$(echo "$PR_BODY" | grep -iE "(npm test|tests?[^|]*[0-9]|[*: ]tests?[*: ]+[0-9])" | grep -oE "[0-9]+/[0-9]+" | head -1)
TICKET_TESTS=$(grep -iE "(npm test|tests?[^|]*[0-9]|[*: ]tests?[*: ]+[0-9])" "$TICKET" | grep -oE "[0-9]+/[0-9]+" | tail -1)
```

**Rationale:**
- `tests?[^|]*[0-9]` matches `Tests 4094`, `Tests: 4094`, `tests passing 4094`, etc. The `[^|]*` prevents accidentally pulling AC count claims like `AC: 14/14` from `| ratio | tests` table cells.
- `[*: ]tests?[*: ]+[0-9]` covers `**Tests**: 4094`, `* Tests: 4094`, `: tests : 4094`.
- Keeps `(npm test)` and `(pass|green)` paths intact for backwards-compatible matches.

**Mode validation:** test against (a) F-H9 PR body `**Tests**: 4094 → 4110`, (b) v0.17.x style `npm test: 1234/1234 PASS`, (c) freeform `tests passing 100%` (must NOT match — no ratio).

### Fix B2 — P2 awk range collapse (`audit-merge.md:65-67`)

**Current:**
```bash
awk '/^## Merge Checklist Evidence/,/^## /' "$TICKET" \
  | grep -E '^\|.*\[x\].*(to be |will |pending|TBD|Will be |to be created|next commit|aspirational)' \
  && flag "P2 drift: aspirational row(s) found"
```

**Problem:** when `## Merge Checklist Evidence` is the last `^## ` H2 in the ticket (true for F-H9, F-H10, and the SDD ticket-template.md scaffold), the awk range-end matches the start line itself; awk emits 1 line (the heading only); grep finds no `[x]` rows. Silent PASS.

**Fix:** flip from awk range to flag-based extraction that runs to EOF when no end-marker matches:
```bash
awk '/^## Merge Checklist Evidence/{flag=1; next} /^## [A-Z]/{flag=0} flag' "$TICKET" \
  | grep -E '^\|.*\[x\].*(to be |will |pending|TBD|Will be |to be created|next commit|aspirational)' \
  && flag "P2 drift: aspirational row(s) found"
```

**Rationale:**
- `flag=1; next` skips the heading itself, so the start-line bug is bypassed.
- `/^## [A-Z]/{flag=0}` resets only on real H2 headings (capital letter after `## `), tolerating `## ` blanks/separators.
- When MCE is the last H2, `flag` stays 1 to EOF — the entire MCE body is emitted.
- BSD-awk compatible (no GNU extensions).

**Verification:** ticket fixture with MCE as last section + `[x]` row containing `will update`.

### Fix B3 — P4 inline branch extraction (`audit-merge.md:85`)

**Current:**
```bash
BRANCH=$(grep -E "^\*\*[Bb]ranch:\*\*" "$TICKET" | head -1 | sed -E 's/^\*\*[Bb]ranch:\*\*[[:space:]]*([^[:space:]|]+).*/\1/')
```

**Problem:** fx ticket headers use a single line `**Status:** Done | **Branch:** feature/F-H9-cat29-seed-expansion (deleted post-merge)`. `^**Branch:**` requires line-start, so the line is rejected.

**Fix:** scan ANY line for the marker and capture the value with `sed -E`:
```bash
BRANCH=$(grep -oE '\*\*[Bb]ranch:\*\*[[:space:]]*[^[:space:]|()]+' "$TICKET" | head -1 | sed -E 's/^\*\*[Bb]ranch:\*\*[[:space:]]*//')
```

**Rationale:**
- `grep -oE` extracts only the matched substring, so the surrounding `**Status:** ... | ` prefix is dropped automatically.
- `[^[:space:]|()]+` stops the branch-name capture at whitespace, pipe, or open-parenthesis (so `(deleted post-merge)` notes do not contaminate).
- Backwards compatible with line-start formats — `grep -oE` matches anywhere on the line, including position 0.

**Verification:** fixture with both inline (`Status | Branch | …`) and dedicated-line (`**Branch:** feature/foo`) header forms. Both must extract `feature/foo` cleanly.

### Fix B4 — P8 Action-column anchor (`audit-merge.md:129`)

**Current:**
```bash
echo "$COMPLETION" | grep -qE "Step[[:space:]]+$step_num([^0-9]|$)" || flag ...
```

**Problem:** the regex matches "Step N" anywhere in any cell. Narrative descriptions of OTHER steps that mention "Step N" satisfy the grep, masking missing dedicated rows.

**Fix:** anchor the regex to the Action column (second `|`-delimited cell) of the Completion Log table:
```bash
echo "$COMPLETION" | grep -qE "^\|[^|]*\|[[:space:]]*Step[[:space:]]+$step_num([^0-9]|$)" \
  || flag "P8 drift: Step $step_num [x] but no dedicated Completion Log entry"
```

**Rationale:**
- The Completion Log convention is `| date | action | notes |`. The Action column is column 2 (between the 1st and 2nd `|` after line-start).
- `^\|[^|]*\|[[:space:]]*Step` requires "Step" at the start of column 2, ignoring narrative content in column 3.
- Preserves the `([^0-9]|$)` end-anchor that prevents "Step 10" from matching for "Step 1".

**Verification:** fixture with both (a) ticket where Step 1 row exists in Action column → must NOT flag; (b) ticket where Step 1 is mentioned only in another step's narrative → must flag.

### Fix B5 — P9 format flexibility (`audit-merge.md:136-137`)

**Current:**
```bash
HEADER_STEP=$(grep -oE 'Step [0-9]+/6' "$TRACKER" | head -1)
DETAIL_STEP=$(grep -A 1 '^\*\*Active Feature:\*\*' "$TRACKER" | grep -oE 'Step [0-9]+/6' | head -1)
```

**Problem:** `Step [0-9]+/6` is a hard requirement, but fx tracker uses `DONE 6/6` (no "Step" prefix). Both empties → silent PASS.

**Fix:** broaden to accept `[0-9]+/6` with optional "Step" prefix:
```bash
HEADER_STEP=$(grep -oE '(Step )?[0-9]+/6' "$TRACKER" | head -1 | sed -E 's/^Step //')
DETAIL_STEP=$(grep -A 1 '^\*\*Active Feature:\*\*' "$TRACKER" | grep -oE '(Step )?[0-9]+/6' | head -1 | sed -E 's/^Step //')
```

**Rationale:**
- Accepts `5/6`, `Step 5/6`, `6/6` etc.
- The post-`sed` strip normalizes both forms, so comparison `[ "$HEADER" = "$DETAIL" ]` is meaningful regardless of which form each side uses.
- If the tracker uses neither form (e.g., `## Status` is the source of truth), values stay empty → conditional skip → no false flag.

**Verification:** fixture with header saying `5/6` and detail saying `Step 5/6` (must NOT flag — same logical state). Fixture with header `5/6` and detail `4/6` (must flag).

### Fix B6 — P5 Status sed harden (`audit-merge.md:94`)

**Current:**
```bash
status=$(grep -E "^\*\*Status:\*\*" "$t" | head -1 | sed -E 's/^\*\*Status:\*\*[[:space:]]*([A-Za-z ]+)[[:space:]]*\|.*/\1/' | sed -E 's/[[:space:]]+$//')
```

**Problem:** when ticket header has `**Status:** **Done**` (bold value embedded inside bold key), the `[A-Za-z ]+` capture group consumes the asterisks and stops at non-letter. Result: `status="**Done**"` literal, comparison `[ "$status" = "Done" ]` fails, ticket erroneously counted as frozen.

Empirical: 56 frozen reported in fx; ~30+ are extraction noise on Done tickets with bold values.

**Fix:** strip leading bold markers before capture, accept multi-word values, and tolerate trailing bold markers:
```bash
status=$(grep -E "^\*\*Status:\*\*" "$t" | head -1 \
  | sed -E 's/^\*\*Status:\*\*[[:space:]]*\*?\*?//' \
  | sed -E 's/[[:space:]]*\*?\*?[[:space:]]*\|.*//' \
  | sed -E 's/[[:space:]]+$//')
```

**Rationale:**
- First sed strips `**Status:**` prefix + optional bold-open `**`.
- Second sed terminates at optional bold-close `**` followed by `|`.
- Third sed trims trailing whitespace.
- Captures `Done`, `**Done**`, `Ready for Merge`, `**In Progress**` correctly to `Done`, `Done`, `Ready for Merge`, `In Progress`.

**Verification:** parametric fixture with status values `Done`, `**Done**`, `Ready for Merge`, `**In Progress** | Branch ... extra`. All must extract clean canonical strings.

### Fix B7 — Execution guardrail (template-level, soft)

**Add at the end of the `### Drift Checks` section (before `### Output Format`):**

```markdown
### Execution discipline

For each of the 11 drift checks (P1–P11), if you declare PASS, **include the literal command output** (or its absence) as evidence in your report. A bare "PASS" without supporting output is treated as **NOT EXECUTED** by the auditor — re-run with output captured.

Recommended pattern:
```
P1 PR body test count stale | PASS | `PR_TESTS=4110/4110, TICKET_TESTS=4110/4110` (matched)
P2 Aspirational rows | PASS | `awk … | grep -E … (no rows matched)`
```

This prevents two failure modes empirically observed: (a) the agent abbreviates execution and reports CLEAN by inference; (b) buggy recipes return empty output silently and the agent treats empty as PASS without verifying the recipe ran. Both are caught when literal output is required.
```

**Rationale:** Documentation-only change. No enforcement automation in v0.18.1 (would be heavy). The directive serves as scaffolding for the agent to follow, and as evidence for the user during cross-check. Out-of-scope for v0.18.1: actually parsing the agent's output for compliance — that requires a hook + parser, future work.
