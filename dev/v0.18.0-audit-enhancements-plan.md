## 3. The 10 patterns — detection specs

Each pattern gets: severity, detection technique (shell commands + regex, reproducible), evidence format, remediation suggestion.

### P1 — PR body test count stale

**Severity:** IMPORTANT
**Detection (v1.1 — rewritten after Gemini G4 + Codex C1):** require the ratio to appear next to a `npm test`/`Tests`/`green`/`pass`/`passing` marker so it binds to test counts not AC/DoD/workflow ratios (14/14, 7/7, 7/8). Use open-ended `[0-9]+/[0-9]+` (works for projects with <1000 tests like sdd-devflow-pb itself: 85/85).
```bash
PR_NUM="<PR>"
PR_BODY=$(gh pr view $PR_NUM --json body -q .body)
TICKET=$(find docs/tickets -iname "*$FEATURE_ID*" | head -1)

# Match test-count ratio specifically (must be near "test"/"tests" marker,
# on the same line, case-insensitive). Open-ended digits.
PR_TESTS=$(echo "$PR_BODY" | grep -iE "(npm test|tests?.*(pass|green)|[0-9]+/[0-9]+.*test)" | \
  grep -oE "[0-9]+/[0-9]+" | head -1)

# Same matcher for ticket; take the LAST occurrence (terminal entry of log)
TICKET_TESTS=$(grep -iE "(npm test|tests?.*(pass|green)|[0-9]+/[0-9]+.*test)" "$TICKET" | \
  grep -oE "[0-9]+/[0-9]+" | tail -1)

[ -n "$PR_TESTS" ] && [ -n "$TICKET_TESTS" ] && [ "$PR_TESTS" != "$TICKET_TESTS" ] && \
  flag IMPORTANT "PR body shows $PR_TESTS, ticket terminal shows $TICKET_TESTS"
```
**Output template:** `PR body test count ($PR_TESTS) diverges from ticket terminal entry ($TICKET_TESTS). Refresh PR body Quality Gates section before merge.`
**Remediation:** grep the PR body for the stale number, replace with the terminal number + delta note (+N new tests).

### P2 — Merge Checklist Evidence rows aspirational

**Severity:** IMPORTANT
**Detection:** Read the `## Merge Checklist Evidence` section of the ticket. For each row marked `[x]`, scan the Evidence column for future-tense phrases:
```bash
awk '/^## Merge Checklist Evidence/,/^## /' "$TICKET" | \
  grep -E '^\| [0-9]+\. ' | \
  grep -E '\[x\].*(to be |will |pending|TBD|Will be |to be created|next commit|aspirational)' \
  && flag IMPORTANT "Row marked [x] but Evidence text is future-tense"
```
**Output template:** Lists each matching row number + the offending phrase.
**Remediation:** rewrite Evidence column with past tense + commit SHA + concrete numbers. Row must reflect reality at time of request-for-merge, not intent.

### P3 — Post-merge actions not enforced in close

**Severity:** IMPORTANT (pre-merge: flag as reminder. post-merge audit: IMPORTANT if not logged)
**Detection (v1.1 — rewritten after Gemini G3 + Codex C2):** collect the textual content (not line-numbered) of unchecked items, then use `grep -Fxq` (fixed-string full-line) for exact match in Completion Log. Avoids the line-number prefix bug and the for-in whitespace split.
```bash
# Extract post-merge items (content only, no line numbers) — use sed to drop "- [ ] " prefix
grep -E "^- \[ \].*(post-merge|post merge|operator|after merge|prod rollout|pending verification)" "$TICKET" \
  | sed -E 's/^- \[ \] //' > /tmp/post_merge_items.txt

# Same from PR body
echo "$PR_BODY" | grep -E "^- \[ \].*(Post-merge|operator)" \
  | sed -E 's/^- \[ \] //' >> /tmp/post_merge_items.txt

# Only check post-merge (ticket Status = Done AND PR state = MERGED).
# Use while-read to handle items with spaces. Use grep -F (fixed string) to
# match as literal substring inside Completion Log, not as regex.
if [ "$TICKET_STATUS" = "Done" ] && [ "$PR_STATE" = "MERGED" ]; then
  COMPLETION_LOG=$(awk '/^## Completion Log/,/^## Merge Checklist/' "$TICKET")
  while IFS= read -r item; do
    [ -z "$item" ] && continue
    # Extract a distinctive noun phrase (first 40 chars) and search for it
    KEY=$(echo "$item" | cut -c1-40)
    echo "$COMPLETION_LOG" | grep -Fq "$KEY" || \
      flag IMPORTANT "Post-merge action not logged: $item"
  done < /tmp/post_merge_items.txt
fi
```
**Output template (pre-merge):** Explicit list of post-merge items the agent must execute + log. Must appear in the "Exact message to agent" section with OBLIGATORIO marker.
**Output template (post-merge audit):** Flag each missing log entry (one line per missing item).

### P4 — Remote branch orphan after "deleted"

**Severity:** NIT (pre-merge, since Step 6 not yet done) · IMPORTANT (post-merge if Workflow Step 6 says [x] branch deleted)
**Detection (v1.1 — rewritten after Gemini G1):** BSD `grep -E` (macOS default) doesn't support `\K`. Use `sed` with a capture group instead.
```bash
BRANCH=$(grep -E "^\*\*[Bb]ranch:\*\*" "$TICKET" | head -1 | \
  sed -E 's/^\*\*[Bb]ranch:\*\*[[:space:]]*([^[:space:]|]+).*/\1/')

# Only check post-merge (when Step 6 should already have run)
if [ "$PR_STATE" = "MERGED" ]; then
  git fetch origin --prune --quiet
  git ls-remote --heads origin "$BRANCH" 2>/dev/null | grep -q refs/heads \
    && flag IMPORTANT "Remote branch $BRANCH still exists despite Workflow Step 6 claim"
fi
```
**Output template:** `Branch $BRANCH still on origin — run: git push origin --delete $BRANCH`

### P5 — Frozen ticket Status

**Severity:** IMPORTANT (single case) · CRITICAL-SYSTEMIC (≥2 frozen in same project)
**Detection (v1.1 — rewritten after Gemini G1 + Codex C3):** `\K\w+` is PCRE-only AND single-word. Real ticket Status values include multi-word "Ready for Merge", "In Progress". Use `sed` capture with broader character class. Also strengthen the merge-detection heuristic: a commit mentioning the ticket ID via `git log --all --grep` is more reliable than basename substring in the oneline stream.
```bash
# Collect frozen tickets: status is anything EXCEPT "Done", and a merge commit
# exists that mentions the ticket ID.
FROZEN=""
for t in $(find docs/tickets -name "*.md"); do
  status=$(grep -E "^\*\*Status:\*\*" "$t" | head -1 | \
    sed -E 's/^\*\*Status:\*\*[[:space:]]*([A-Za-z ]+)[[:space:]]*\|.*/\1/' | \
    sed -E 's/[[:space:]]+$//')
  [ "$status" = "Done" ] && continue
  ticket_id=$(basename "$t" .md | sed -E 's/-[a-z].*//')
  # Look for a merge commit that references the ticket id (`<ID>` or `(#NNN)`)
  if git log --all --oneline --grep="$ticket_id" | grep -q .; then
    FROZEN="$FROZEN $t"
  fi
done
FROZEN_COUNT=$(echo "$FROZEN" | tr ' ' '\n' | grep -c '.')

if [ "$FROZEN_COUNT" -eq 1 ]; then
  flag IMPORTANT "1 frozen ticket: $FROZEN"
elif [ "$FROZEN_COUNT" -ge 2 ]; then
  flag CRITICAL-SYSTEMIC "$FROZEN_COUNT frozen tickets detected — systemic close-pattern gap:$FROZEN"
fi
```
**Note**: the merge-detection via `git log --grep` is still heuristic (a ticket could be mentioned in a non-merge commit). Acceptable for v0.18.0; tightening to "only merge-commits on main/develop via `--merges` and `--first-parent`" is a v0.18.x follow-up.

**Severity:** NIT

```bash
AC_SECTION=$(awk '/^## Acceptance Criteria/,/^## /' "$TICKET")

```
**Output template:** Compare actual vs claimed. Report discrepancy.

### P7 — Test count drift **within** ticket (final sections only)

**Severity:** IMPORTANT
**Detection (v1.1 — rewritten after Codex C4 contradiction finding):** the previous naive "≥2 distinct numbers anywhere" over-fires because Completion Log legitimately contains intermediate-state numbers ("Step 3 Plan Step 8: 3694/3694"). Narrow to the **final-state sections**: AC section, DoD section, and tracker Active Session entry. Use `[0-9]+/[0-9]+` (open-ended digits).
```bash
# Extract terminal count from Completion Log (last row's X/Y ratio near "test"/"tests"/"pass")
TERMINAL=$(awk '/^## Completion Log/,/^## Merge Checklist/' "$TICKET" | \
  grep -iE "(test|pass|green)" | grep -oE "[0-9]+/[0-9]+" | tail -1)

# Gather final-state numbers from AC + DoD sections
AC_SECTION=$(awk '/^## Acceptance Criteria/,/^## Definition of Done/' "$TICKET")
DOD_SECTION=$(awk '/^## Definition of Done/,/^## Workflow Checklist/' "$TICKET")
FINAL_NUMS=$(printf '%s\n%s\n' "$AC_SECTION" "$DOD_SECTION" | \
  grep -iE "(test|pass|green)" | grep -oE "[0-9]+/[0-9]+" | sort -u)

# Tracker Active Session number (if present)
TRACKER=$(awk '/^\*\*Active Feature:\*\*/{flag=1} flag' docs/project_notes/product-tracker.md 2>/dev/null | \
  head -5 | grep -iE "(test|pass|green)" | grep -oE "[0-9]+/[0-9]+" | head -1)

# Any number in final-state sections that differs from TERMINAL = drift.
for n in $FINAL_NUMS $TRACKER; do
  [ -n "$TERMINAL" ] && [ "$n" != "$TERMINAL" ] && \
    flag IMPORTANT "Final-section count $n diverges from Completion Log terminal $TERMINAL"
done
```
**Output template:** Per divergent occurrence: `AC13 shows $n — Completion Log terminal shows $TERMINAL. Delta: $((TERMINAL - n)) tests.`
**Note:** Intermediate rows in Completion Log (e.g., "Step 3 Plan Step 8: 3694/3694" followed by "Step 5 gates post-fixes: 3723/3723") are legitimate snapshots of different moments and do NOT fire the detection. Only final-state sections count.

### P8 — Completion Log gap vs Workflow Checklist

**Severity:** IMPORTANT
**Detection (v1.1 — rewritten after Gemini G2 + Codex C5):** the previous `for-in $(...)` splits `grep -oE` output by whitespace, so "Step 1: Plan" became 3 iterations. New recipe uses `while-read` over unique step numbers from **only checked** workflow items. Note: Step 5 is commonly split into 2 sub-rows (code-review + qa-engineer) — count unique step numbers, not unique lines.
```bash
WORKFLOW=$(awk '/^## Workflow Checklist/,/^## Completion Log/' "$TICKET")
COMPLETION=$(awk '/^## Completion Log/,/^## Merge Checklist/' "$TICKET")

# Unique step numbers from CHECKED workflow items only
# Pattern: "- [x] Step N: ..." → capture N
CHECKED_STEPS=$(echo "$WORKFLOW" | grep -E "^- \[x\] Step [0-9]+:" | \
  sed -E 's/^- \[x\] Step ([0-9]+):.*/\1/' | sort -u)

# For each checked step number, require ≥1 Completion Log row mentioning "Step N"
while read -r step_num; do
  [ -z "$step_num" ] && continue
  echo "$COMPLETION" | grep -qE "Step[[:space:]]+$step_num([^0-9]|$)" || \
    flag IMPORTANT "Workflow Step $step_num marked [x] but no Completion Log narrative"
done <<< "$CHECKED_STEPS"
```
**Output template:** One line per `[x]` step without corresponding log narrative: `Step N marked [x] but no Completion Log entry. Add row summarizing: agent X, verdict Y, commit Z.`
**Edge case:** Step 5 split across 2 rows (code-review + qa-engineer) — the unique step number `5` fires once, so if both subitems are `[x]` and Completion Log has ≥1 row mentioning "Step 5", detection passes. A finer check (both code-review AND qa-engineer logged separately) is deferred to v0.18.x.

### P11 — Tracker Features table status vs ticket Status mismatch (added v1.1, Gemini G6)

**Severity:** IMPORTANT
**Context:** distinct from P9 (P9 = tracker *header* Last Updated line stale vs Active Session detail). P11 = tracker Features table row status (`in-progress`/`done`) differs from ticket header Status (`Ready for Merge`/`Done`).
**Detection:**
```bash
FEATURE_ID="<passed to skill>"
TICKET_STATUS=$(grep -E "^\*\*Status:\*\*" "$TICKET" | head -1 | \
  sed -E 's/^\*\*Status:\*\*[[:space:]]*([A-Za-z ]+)[[:space:]]*\|.*/\1/' | \
  sed -E 's/[[:space:]]+$//')

TRACKER_ROW=$(grep -F "$FEATURE_ID" docs/project_notes/product-tracker.md | \
  grep -E "\| in-progress \||\| done \|\|\| pending \|" | head -1)
TRACKER_STATUS=$(echo "$TRACKER_ROW" | grep -oE "\| (in-progress|done|pending|blocked) \|" | head -1 | \
  sed -E 's/\| ([a-z-]+) \|/\1/')

# Mapping: expected tracker status per ticket header
case "$TICKET_STATUS" in
  "Ready for Merge"|"Review") EXPECTED="in-progress" ;;
  "Done") EXPECTED="done" ;;
  "In Progress"|"Planning"|"Spec") EXPECTED="in-progress" ;;
  *) EXPECTED="" ;;
esac

[ -n "$EXPECTED" ] && [ -n "$TRACKER_STATUS" ] && [ "$TRACKER_STATUS" != "$EXPECTED" ] && \
  flag IMPORTANT "Ticket Status=\"$TICKET_STATUS\" expects tracker=$EXPECTED but tracker shows=\"$TRACKER_STATUS\""
```
**Output template:** `Ticket header says $TICKET_STATUS → Features table row should say $EXPECTED (currently $TRACKER_STATUS). Update product-tracker.md.`

### P10 — Duplicate Completion Log rows

**Severity:** NIT
**Detection (v1.1 — tightened after Codex C7):** 30-char prefix was too loose — "Updated tracker..." is a legitimate repeated action across sessions. Use full-row hash (date + action + first 80 chars of notes) to keep noise down.
```bash
awk -F'|' '/^\| [0-9]{4}-[0-9]{2}-[0-9]{2}/ {
  # Columns: | | date | action | notes |
  key = $2 "|" $3 "|" substr($4, 1, 80)
  gsub(/^[[:space:]]+|[[:space:]]+$/, "", key)
  print key
}' "$TICKET" | sort | uniq -d
```
**Output template:** Lists duplicated `date | action | notes-prefix` triples. Blank output = no duplicates. Non-blank = dupes suspected.
**Note:** `uniq -d` prints only lines that appear ≥2 times; legitimate similar actions on different dates still have distinct `date|action|notes` keys and don't fire.

- Fix suggestions gets per-pattern remediation.

**Parallel update:** `template/.gemini/commands/audit-merge-instructions.md` gets the same 10 checks in its parallel format.

**Key design decision — SEPARATE verdicts for structural vs drift (v1.1 — post-Codex C6):**
The existing skill's single PASS/FAIL verdict at the bottom doesn't cleanly accommodate "11 structural checks passed, but 3 drift checks flagged". Codex noted the ambiguity. v1.1 resolution: emit **two distinct verdicts**, one per category.

- **Structural verdict (Checks 1-11)**: `STRUCTURAL: READY FOR MERGE` (all PASS) or `STRUCTURAL: NEEDS FIX — N blockers`. These are blocking.
- **Drift verdict (Checks 12-22)**: `DRIFT: CLEAN` (all 12-22 PASS) or `DRIFT: N advisories — refresh before merge`. These are **non-blocking for the merge gate but MUST be addressed before requesting user authorization** (per SDD discipline the user will catch them anyway).

Combined verdict row at the bottom:
- Both PASS → `READY FOR MERGE (compliance 11/11, drift 0/N advisories)`
- Structural fail → `NEEDS FIX — N structural blockers (+ any drift noted)`
- Structural pass + drift advisories → `READY FOR MERGE PENDING DRIFT CLEANUP (N advisories)`

**Rationale:** the current `/audit-merge` is agent-facing self-audit. Hard-failing 10 new checks could loop an agent into auto-fix cycles if the agent mis-interprets the remediation. Separate verdict keeps the blocking-vs-advisory signal clear without mixing semantics.

## 9. Risks

- **R1 Over-detection on legitimate intermediate states.** E.g., during Step 3 TDD, the ticket Completion Log might have "3668 → 3694" then later "3700 → 3723" and both are valid snapshots. Mitigation: P7 explicitly says "only IMPORTANT if final AC/DoD/tracker differ from Completion Log *terminal* entry". "Terminal" = **last row** of the `## Completion Log` table (highest date + lowest row-position for same date). Intermediate rows are allowed to disagree.
- **R2 Regex brittleness on non-English tickets.** The detection regexes assume English keywords ("will ", "to be ", "post-merge"). fx mixes Spanish in Active Session entries but English in ticket templates. Mitigation: keep regexes English-only for now; add Spanish variants in v0.18.1 if user base demands. Document in skill file.
- **R3 Skill-file too long.** At ~175 + ~140 lines, both skills remain well under 500 (a rough readability threshold). No splitting into sub-files.
- **R4 Shell commands in skill markdown are decorative, not executable by the runtime.** Claude parses and interprets them as guidance. The LLM translates to actual bash/grep calls. This is the existing pattern (pre-0.18.0 commands look the same). No change.
- **R5 GH API rate limit** — detection P1 and P2 call `gh pr view` once per audit (one call for body, already-retrieved metadata reused). Additional calls NOT introduced. `git` local operations (diff, ls-remote, log) have no rate limit. Stays within the existing shared 5000/hr GitHub budget.
- **R6 audit-merge.md NOT in smart-diff provenance** — `audit-merge.md` is NOT in `expectedSmartDiffTrackedPaths` (lib/meta.js:244). Upgrades from 0.17.x → 0.18.0 will wholesale-overwrite any local customizations the user made to `audit-merge.md`. **Pragmatic accepted risk**: pre-0.18.0 was already wholesale-recopy for this file (no regression), and `audit-merge.md` is a command skill whose customization is rare. Documented as known limitation; adding audit-merge to smart-diff is deferred to v0.18.1+ (part of the "Full smart-diff coverage for ~30 remaining template files" item in ROADMAP). A one-line mention lands in the CHANGELOG upgrade-notes section.
- **R7 False-positives on P6** — "all 24 marked" text claim can be stale by 2 (like F-H4-B). But some tickets have nested AC groups (AC-1a..g + AC-2a..c etc.) where the "total" is genuinely ambiguous. Detection should flag only when the claimed number diverges by ≥2 from both actual `[x]` count AND the sum of `[x]+[ ]`. Prevents noise on partial-state tickets.
