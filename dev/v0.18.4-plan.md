## 1. Executive summary

v0.18.4 closes two empirically-surfaced gaps in `--upgrade` and `/audit-merge`:

- **G1 (`lib/upgrade-generator.js` fallback PRESERVE path)**: when the fallback content-compare (Case 3c) decides to preserve user content + write `.new`, NO hash is recorded in `.sdd-meta.json`. Consequence: same file falls to fallback Case 3 on every future upgrade, regenerating `.new` indefinitely. fx today has 7 affected paths: 2 non-customized (audit-merge.md ×2 — accepted manually) + 5 legitimately-customized (AGENTS.md + 4 planners — rejected manually). Fix: on fallback PRESERVE, record the **template's hash** (not the user's) so next upgrade enters Case 2 hash-based path; Case 2a (user accepted .new) replaces cleanly, Case 2b (user kept customization) preserves without re-running fallback.
- **P11-B (`template/.claude/commands/audit-merge.md` P11 recipe + Gemini twin)**: tickets matching the `-lite` / `-FU` / `-FU[0-9]*` sub-scope suffix pattern (uppercase-only — lowercase `-fu*` deliberately not shipped per R2 Gemini M1) have FEATURE_ID stripped to the parent feature ID (e.g. `F116-lite-ci-hardening.md` → `F116`), and P11 then compares the sub-scope ticket's Status against the parent feature's tracker-row status, which is **intentionally** different (parent stays `pending`/`in-progress` while sub-scope ticket reaches `Done`). Fix: detect the sub-scope suffix on the **original basename** before stripping, emit `P11 N/A: $BASENAME is sub-scope ticket — parent tracker row status independent`, and skip the comparison.

Both fixes preserve the **Codex M1 invariant** ("preserved files in Case 2b do NOT get a new hash") because G1 only mutates `newHashes` when the path has **no prior hash entry** (Case 3c → Case 2 bootstrap). Plan v0.1 derives the design from re-reading `lib/upgrade-generator.js:438-544` (agents fallback) + `:799-839` (standards fallback) and the v0.17.0 Codex M1 rationale.

### 2.2 P11-B — fx F116-lite-ci-hardening ticket cohort

`docs/tickets/F116-lite-ci-hardening.md`:
```
**Status:** Done (squash merged 2026-05-13)
```

`docs/project_notes/product-tracker.md` Features-Backend table:
```
| F116 | API + scraper lint cleanup + CI hardening | pending |
```

`/audit-merge` execution today on this ticket:
1. `FEATURE_ID=$(basename "$TICKET" .md | sed -E 's/-[a-z].*//')` → `F116`
2. `TICKET_STATUS=Done` → `EXPECTED=done`
3. `grep -F "F116" tracker | grep -oE "\| (in-progress|done|pending|blocked) \|"` → returns `| pending |` (parent's row)
4. P11 fires: `P11 drift: ticket Status='Done' expects tracker='done' but tracker='pending'`

This is a **false positive**: F116 parent feature still has deferred items (frontend lint cleanup, monorepo build-filters), so its tracker row correctly stays `pending`. The `-lite` ticket closes a sub-scope (the API lint + CI hardening portion) and is correctly `Done`. The two are **architecturally decoupled**.

**fx convention** (also documented in fx's `docs/project_notes/decisions.md`): `-lite` = sub-scope, `-FU` = follow-up, `-fu1`/`-fu2`/... = numbered follow-ups. All three patterns split a parent feature into independently-closeable units.

**Why P11 didn't fire empirically today**: the `grep -F` returns the first match, which may be a narrative mention earlier in the tracker (e.g. "F116 parent depends on F-H10") rather than the table row. The `grep -oE "\| (in-progress|done|pending|blocked) \|"` then matches the first status pipe-cell on that narrative line — could be ANY feature's status. The "protection" is accidental and breaks the moment the tracker is reordered.

**Implementation approach (post-Codex M2)**: inline `newHashes[…]` write **after** each Case 3c call site, NOT inside the helpers. Preserves the helpers' Codex M1 semantics for Case 2b callers; opt-in bootstrap is explicit at each call site:

```js
// Site #1 — agents Case 3c (line 544):
preserveFile(adaptedFullTargetFallback);
newHashes[posixPath] = computeHash(adaptedFullTargetFallback);  // v0.18.4 G1
continue;

// Site #2 — commands Case 3c (line 654):
preserveCmd();
newHashes[posixPath] = computeHash(rawTemplate);  // v0.18.4 G1
// (no `continue` needed — already at end of for-iteration)

// Site #3 — workflow-core Case 3c (line 822-838, inline):
// after the existing `preserved++; continue;` block, add:
newHashes[posix] = computeHash(adaptedTarget);  // v0.18.4 G1

// Site #4 — standards Case 3c (line 982, via helper):
// add after the helper call, mirroring site #1 pattern:
newHashes[spec.posix] = computeHash(adaptedTarget);  // v0.18.4 G1

// Site #5 — AGENTS.md Case 3c (line 1057):
preserveAgentsMd();
newHashes[AGENTS_MD_POSIX] = computeHash(adaptedAgentsMd);  // v0.18.4 G1
```

**Rationale for recording the TEMPLATE hash (not the user's hash)**: when the user accepts the `.new` content on a later reconciliation, their file content becomes equal to the template content. Next upgrade computes `currentHash` == `storedHash` (the template hash we recorded), enters Case 2a (pristine match), and replaces cleanly with the next-version template. If we'd recorded the USER's customized hash instead, the same scenario would also enter Case 2a — but interpreted as "user has the pristine baseline", which would SILENTLY overwrite the customization if the user reverted. Recording the template hash is the correct semantic anchor.

**Risk (post-M2 refinement)**: still possible to misclassify a Case 2b call site as Case 3c. Mitigation: AC2 + cross-model R3 specifically audit each call site against its dispatch branch. The site inventory table above is the source of truth.

### Phase 2 — P11-B sub-scope ticket detection

**Files**:
- `template/.claude/commands/audit-merge.md` — P11 recipe at line 181-198
- `template/.gemini/commands/audit-merge-instructions.md` — Gemini twin (mirror parity)

**Diff** (Claude side; Gemini byte-equal):
```bash
TICKET_STATUS=$(grep -E "^\*\*Status:\*\*" "$TICKET" | head -1 \
    | sed -E 's/^\*\*Status:\*\*[[:space:]]*\*?\*?//' \
    | sed -E 's/[[:space:]]*\*?\*?[[:space:]]*\|.*//' \
    | sed -E 's/[[:space:]]+(\(.*\)|—.*|–.*|-.*)$//' \
    | sed -E 's/\*\*[[:space:]]*$//' \
    | sed -E 's/[[:space:]]+$//')
TICKET_BASENAME=$(basename "$TICKET" .md)
# v0.18.4 P11-B: sub-scope tickets (-lite / -FU / -fu*) close a partial scope
# of a parent feature; the parent tracker row stays at its parent status
# (typically `pending` or `in-progress`) while the sub-scope ticket reaches
# `Done`. P11 must NOT enforce status mapping across this boundary.
case "$TICKET_BASENAME" in
  *-lite|*-lite-*|*-FU|*-FU-*|*-FU[0-9]*)
    echo "P11 N/A: $TICKET_BASENAME is a sub-scope ticket — parent tracker row status independent"
    ;;
  *)
    FEATURE_ID=$(echo "$TICKET_BASENAME" | sed -E 's/-[a-z].*//')
    TRACKER_STATUS=$(grep -E "^\|[[:space:]]*$FEATURE_ID[[:space:]]*\|" docs/project_notes/product-tracker.md \
        | grep -oE "\| (in-progress|done|pending|blocked) \|" | head -1 | sed -E 's/\| ([a-z-]+) \|/\1/')
    case "$TICKET_STATUS" in
      "Ready for Merge"|"Review"|"In Progress"|"Planning"|"Spec") EXPECTED="in-progress" ;;
      "Done") EXPECTED="done" ;;
      *) EXPECTED="" ;;
    esac
    [ -n "$EXPECTED" ] && [ -n "$TRACKER_STATUS" ] && [ "$TRACKER_STATUS" != "$EXPECTED" ] \
      && flag "P11 drift: ticket Status='$TICKET_STATUS' expects tracker='$EXPECTED' but tracker='$TRACKER_STATUS'"
    ;;
esac
```

**Two changes folded in**:
1. **Sub-scope branch**: case statement emits `P11 N/A` and skips comparison.
2. **Drive-by P11 hardening** (related, low-risk): replaced `grep -F "$FEATURE_ID"` with `grep -E "^\|[[:space:]]*$FEATURE_ID[[:space:]]*\|"` so the tracker lookup requires the FEATURE_ID to be the first pipe-table cell, matching the v0.18.3 P16 hardening idiom. This makes the "regular ticket" branch robust against narrative mentions ordering before the table row.

**Mirror parity**: scenario #108 auto-validates that ALL bash blocks are byte-equal Claude ↔ Gemini.
